Chapter 00 — The Three Roles: ML Engineer vs MLOps Engineer vs AI Engineer

Before any technical prep, you must know which role you're interviewing for, because the same résumé gets evaluated against very different bars. Interviewers calibrate questions to the role. Answering an MLOps system-design question with a pure modeling answer (or vice versa) is the single most common reason strong candidates get "not a fit."

1. Machine Learning Engineer (MLE)

Mandate: turn a business/data problem into a model that measurably performs, and make that model good enough and fast enough to use.

Day to day:

  • Frame the problem (classification? ranking? regression? what's the label?).
  • Build features, handle leakage, split data correctly.
  • Train, tune, and evaluate models; reason about bias/variance.
  • Get the model to acceptable latency/size (quantization, distillation).
  • Hand a model artifact + metrics to the MLOps/serving layer.

Interviewed on: ML fundamentals (bias-variance, regularization, metrics), classical ML (Phase 2a), deep learning (Phase 2b), a coding round, and an ML-design round ("design a fraud detector"). Math matters: you will be asked how gradient descent, a decision tree split, or backprop works.

Maps to: Phases 1, 2, 6 · Certs: AWS MLA-C01, GCP Professional ML Engineer.

2. MLOps Engineer

Mandate: make ML reproducible, deployable, observable, and reliable. You own the path from "a notebook that works on my laptop" to "a service the company depends on," and everything that keeps it healthy after launch.

Day to day:

  • Build training/inference pipelines (Airflow/Kubeflow/Databricks).
  • CI/CD for models: versioning data + models (DVC, MLflow), automated retraining.
  • Containerize and deploy on Kubernetes; manage infra as code (Terraform).
  • Monitor for data drift, concept drift, latency, cost; set up rollback.
  • Own the model registry, feature store, and reproducibility guarantees.

Interviewed on: system design ("design a feature store", "how do you do a zero-downtime model rollout?"), Docker/K8s, CI/CD, monitoring, infra, plus enough ML to reason about what can break (drift, training/serving skew). Less math, more systems.

Maps to: Phases 5, 7, 8, 9, 10 · Certs: Databricks MLOps, CKA, Terraform, NVIDIA NCA-AIIO.

3. AI Engineer (a.k.a. GenAI / LLM Engineer)

Mandate: build applications on top of foundation models. You usually don't train the base model — you make it useful, grounded, safe, and cheap via prompting, retrieval, tools, and orchestration.

Day to day:

  • Design prompts and prompt pipelines; pick models for cost/quality/latency.
  • Build RAG systems (chunking, embeddings, vector DBs, reranking).
  • Build agents (tool use, planning, memory, multi-step workflows).
  • Evaluate non-deterministic output (LLM-as-judge, golden sets, guardrails).
  • Sometimes fine-tune (LoRA) when prompting/RAG isn't enough.

Interviewed on: LLM fundamentals (tokens, context window, temperature, attention at a high level), RAG architecture, agent design, evaluation of non-deterministic systems, hallucination mitigation, cost/latency tradeoffs.

Maps to: Phases 3, 4 · Certs: IBM watsonx GenAI, NVIDIA GenAI & LLMs, Azure AI Apps & Agents, GenAI with LLMs (DeepLearning.AI).

Chapter 01 (Phase 1) — Software Engineering Foundations

Every ML/AI/MLOps loop has a coding + fundamentals round, and it is the most common silent rejection: brilliant modeling, but couldn't reason about the GIL, write clean Python, or explain what docker build does. This chapter covers the Python data model, concurrency, Git internals, Docker layers, and DS&A — at the same code-level depth as the scikit-learn chapter.

1. The Python data model — "everything is an object"

Concept map: Python's behavior is defined by dunder (double-underscore) methods. len(x) calls x.__len__(); x + y calls x.__add__(y); x[i] calls x.__getitem__(i). Operators and built-ins are syntactic sugar over them.

class Vector:
    def __init__(self, *components):
        self.components = list(components)
    def __len__(self):                       # len(v)
        return len(self.components)
    def __getitem__(self, i):                # v[i], also makes it iterable
        return self.components[i]
    def __add__(self, other):                # v1 + v2
        return Vector(*(a + b for a, b in zip(self, other)))
    def __repr__(self):                      # repr(v), debugging
        return f"Vector{tuple(self.components)}"

v = Vector(1, 2, 3)
len(v)          # 3   -> __len__
v[0]            # 1   -> __getitem__
v + Vector(4, 5, 6)   # Vector(5, 7, 9) -> __add__

Interview gold: "How would you make a class iterable?" → implement __iter__ (or just __getitem__ from 0). "Sortable?" → __lt__ + @total_ordering, or pass key= to sorted.

Mutable vs immutable & the default-argument trap

def append_to(x, items=[]):     # BUG: default list created ONCE at def time
    items.append(x)
    return items
append_to(1)   # [1]
append_to(2)   # [1, 2]  <-- shared across calls!

def append_to(x, items=None):   # FIX
    items = items if items is not None else []
    items.append(x)
    return items

The classic trap. Default arguments are evaluated once, at function definition, not per call. Mutable defaults leak state between calls.

2. The GIL & concurrency — the #1 Python systems question

The Global Interpreter Lock is a mutex that lets only one thread execute Python bytecode at a time in CPython. Consequence:

Workload Right tool Why
CPU-bound (matrix math in pure Python, parsing) multiprocessing separate processes = separate GILs = true parallelism
I/O-bound (HTTP calls, DB, disk) threading or asyncio the GIL is released during I/O waits, so threads overlap
Many concurrent I/O (1000s of API calls) asyncio one thread, cooperative await, no thread overhead 
# I/O-bound: asyncio overlaps 100 LLM calls without 100 threads
import asyncio, aiohttp

async def fetch(session, url):
    async with session.get(url) as r:
        return await r.json()          # await yields control during the wait

async def main(urls):
    async with aiohttp.ClientSession() as s:
        return await asyncio.gather(*(fetch(s, u) for u in urls))

Interview trap: "Does threading speed up a CPU-bound loop in Python?" → No — the GIL serializes bytecode; you get concurrency, not parallelism. Use multiprocessing or push the compute into NumPy/C (which releases the GIL). Why does NumPy escape the GIL? Its heavy loops run in C and release the GIL.

3. Generators & memory — lazy evaluation

def read_large_file(path):
    with open(path) as f:
        for line in f:          # file objects are lazy iterators
            yield line.strip()  # yield -> generator; one line in memory at a time

# vs the memory bomb:
lines = open(path).readlines()  # loads the WHOLE file into RAM

yield turns a function into a generator: it produces values one at a time, pausing/resuming, holding O(1) memory instead of O(n). Critical for ML data loaders streaming GBs that don't fit in RAM.

Q: range(10**9) — does it allocate a billion ints? → No, range is lazy. list(range(10**9)) would OOM. Same idea as generators.

4. Git internals — what commands actually do

Concept map: Git is a content-addressed store of commits (snapshots, each pointing to its parent(s)), trees (directories), and blobs (file contents). A branch is just a movable pointer to a commit; HEAD points to your current branch.

merge vs rebase (the question everyone gets)

# MERGE: preserves history, creates a merge commit joining two lineages
git checkout main
git merge feature        # new commit M with two parents

# REBASE: REWRITES history — replays your commits on top of main's tip
git checkout feature
git rebase main          # feature's commits get NEW hashes, linear history
  • Rebase gives a clean linear history but rewrites commit hashes → never rebase a branch others have pulled (you diverge their history).
  • Golden rule: rebase local/private branches; merge shared ones.
git reset --soft HEAD~1    # undo commit, KEEP changes staged
git reset --hard HEAD~1    # undo commit, DISCARD changes (dangerous)
git revert <sha>           # NEW commit that undoes <sha> (safe on shared branches)
git cherry-pick <sha>      # copy one commit onto current branch

Q: "You committed a secret. How do you remove it?" → It's in history, so a new commit isn't enough. git filter-repo (or BFG) to rewrite history, force-push, and rotate the secret — assume it's compromised. (This exact scenario came up in this very project's handover notes.)

5. Docker — what docker build does, layer by layer

Concept map: A Docker image is a stack of read-only layers; each Dockerfile instruction (RUN, COPY, ADD) creates one layer. A container is a running image plus a thin writable layer on top.

FROM python:3.11-slim                 # layer 0: base image
WORKDIR /app
COPY requirements.txt .               # layer: just the deps file
RUN pip install -r requirements.txt   # layer: installed packages  <- expensive
COPY . .                              # layer: your source code     <- changes often
CMD ["python", "serve.py"]            # metadata: default process

Why COPY requirements.txt BEFORE COPY . .? — layer caching. Docker reuses a cached layer if its inputs are unchanged. By copying deps first, editing your source only busts the cheap last layer, not the expensive pip install. Reverse the order and every code change reinstalls all packages.

docker build -t myapp:1.0 .     # reads Dockerfile, builds layers
docker run -p 8000:8000 myapp:1.0   # host:container port mapping
docker exec -it <id> /bin/sh    # shell into a running container

Q: image vs container? → Image = immutable blueprint (the class); container = running instance (the object). Q: how shrink an image? → slim/alpine base, multi-stage builds, .dockerignore, combine RUNs to reduce layers. Q: VM vs container? → Containers share the host kernel (lightweight, seconds to start); VMs virtualize hardware with a full guest OS (heavy, minutes).

6. DS&A — the patterns that actually show up

You won't implement red-black trees. You will get hash-map counting, two-pointer/sliding-window, and "what's the complexity?".

from collections import Counter, defaultdict, deque

# Hash map — O(n) frequency, the workhorse of ML preprocessing
Counter("mississippi")          # {'i':4,'s':4,'p':2,'m':1}

# Two pointers — O(n), O(1) space
def two_sum_sorted(a, target):
    i, j = 0, len(a) - 1
    while i < j:
        s = a[i] + a[j]
        if s == target: return (i, j)
        if s < target: i += 1
        else: j -= 1

# Sliding window — longest substring without repeats, O(n)
def longest_unique(s):
    seen, left, best = {}, 0, 0
    for right, ch in enumerate(s):
        if ch in seen and seen[ch] >= left:
            left = seen[ch] + 1
        seen[ch] = right
        best = max(best, right - left + 1)
    return best

Big-O cheat sheet

Structure / op Average Note
dict/set lookup O(1) hash collision worst case O(n)
list index O(1) append amortized O(1)
list insert/delete front O(n) use deque for O(1) ends
sort O(n log n) Timsort
binary search (sorted) O(log n) bisect module

Q: When is a dict O(n)? → pathological hash collisions. Q: list vs deque? → deque gives O(1) appendleft/popleft; a list's front ops are O(n).

7. pandas / NumPy — the data-wrangling round

import numpy as np, pandas as pd

# Vectorize — NEVER loop rows in pandas
df["z"] = (df["x"] - df["x"].mean()) / df["x"].std()   # vectorized, C-speed
# anti-pattern: df.apply(lambda r: ..., axis=1)  # 100x slower, Python-level loop

# Broadcasting — NumPy aligns shapes without copying
A = np.ones((3, 1));  b = np.array([10, 20, 30])   # (3,1) + (3,) -> (3,3)

df.groupby("city")["price"].agg(["mean", "count"])  # split-apply-combine
df.merge(other, on="id", how="left")                # SQL-style join

Q: why avoid iterrows/apply(axis=1)? → they run a Python-level loop per row, bypassing the vectorized C engine — often 10–100× slower. Q: loc vs iloc? → loc = label-based, iloc = integer-position-based.

8. Interview Q&A (rapid fire)

  1. Q: is vs ==? → is compares identity (same object); == compares value (__eq__). Don't use is for value checks (except is None).
  2. Q: shallow vs deep copy? → shallow copies the container but shares nested objects; copy.deepcopy recurses. Matters for nested configs/lists.
  3. Q: decorator? → a function that wraps another to add behavior (@lru_cache, @property, @staticmethod). It's f = decorator(f).
  4. Q: context manager / with? → __enter__/__exit__ guarantee cleanup (file/connection close) even on exception.
  5. Q: list vs tuple? → list mutable, tuple immutable + hashable (usable as dict keys); tuples signal fixed-size records.
  6. Q: how does Python manage memory? → reference counting + a cyclic garbage collector for reference cycles.
  7. Q: *args / **kwargs? → variadic positional / keyword arguments.

Chapter 02 (Phase 2a) — Classical ML & scikit-learn: How does the model learn?

This is the depth benchmark chapter. The recurring interview question is not "what is a random forest?" — it's "walk me through what happens when you call .fit()". This chapter answers that at the code-and-math level for every major scikit-learn estimator.

0. The one abstraction that unifies all of scikit-learn: the Estimator API

Every model in scikit-learn — linear regression, an SVM, a whole pipeline — implements the same three-method contract. If you understand this, you understand the library.

estimator.fit(X, y)        # LEARN: estimate parameters from training data
estimator.predict(X_new)   # APPLY: use learned parameters to produce outputs
estimator.score(X, y)      # EVALUATE: default metric (R² for regressors,
                           #           accuracy for classifiers)
  • X is shape (n_samples, n_features) — a 2D array/DataFrame.
  • y is shape (n_samples,) — the target/label vector.
  • Transformers (scalers, encoders) implement fit + transform instead of predict; fit_transform does both in one pass.
  • Classifiers additionally implement predict_proba (class probabilities) and sometimes decision_function (raw scores/margins).

The golden convention: learned attributes end in _

After fit(), everything the model learned is stored on the object with a trailing underscore. This is the single most useful thing to know:

from sklearn.linear_model import LinearRegression
model = LinearRegression()
model.fit(X, y)

model.coef_        # learned weights  (one per feature)   <- learned, has _
model.intercept_   # learned bias term                    <- learned, has _
model.n_features_in_  # how many features fit() saw

vs. hyperparameters (which you set, the model does NOT learn) — no underscore:

model = Ridge(alpha=1.0)   # alpha is a hyperparameter you chose
model.alpha                # 1.0  (no underscore — you set it)

Interview gold: "How do you tell what a scikit-learn model learned?" → "Inspect the trailing-underscore attributes — coef_, intercept_, feature_importances_, cluster_centers_, etc. They only exist after fit."

1. LinearRegression — learning by a closed-form equation (no iteration!)

What it learns: weights w and bias b so that ŷ = Xw + b minimizes the mean squared error MSE = (1/n) Σ (yᵢ − ŷᵢ)².

How .fit() learns it: not gradient descent. Ordinary Least Squares has a closed-form analytical solution — the Normal Equation:

w = (XᵀX)⁻¹ Xᵀ y

scikit-learn actually solves it via an SVD-based least-squares routine (scipy.linalg.lstsq) for numerical stability, but conceptually it's "solve the equation in one shot." There is no learning rate and no epochs.

from sklearn.linear_model import LinearRegression
import numpy as np

X = np.array([[1], [2], [3], [4]])   # 4 samples, 1 feature
y = np.array([2, 4, 6, 8])           # perfectly y = 2x

model = LinearRegression()
model.fit(X, y)                      # <-- solves the normal equation

print(model.coef_)        # [2.]   learned slope
print(model.intercept_)   # ~0.0   learned bias
print(model.predict([[10]]))  # [20.]  applies ŷ = 2*10 + 0

Interview follow-ups:

  • "Why not always use the closed form?" → (XᵀX)⁻¹ is O(n_features³) to invert and needs the matrix to be invertible (fails with collinear features). For many features or huge data, iterative gradient methods scale better → that's SGDRegressor.
  • "What breaks it?" → multicollinearity (near-singular XᵀX) → unstable, huge weights → fix with Ridge (L2) which adds αI to make it invertible.

2. Ridge / Lasso — same model, regularized learning

Identical prediction form (ŷ = Xw + b), different loss that .fit() minimizes:

Model Minimizes Effect of penalty
LinearRegression MSE none
Ridge (L2) MSE + α·Σwⱼ² shrinks weights toward 0 (never exactly 0)
Lasso (L1) MSE + α·Σ|wⱼ| drives some weights exactly to 0 → feature selection
  • Ridge still has a closed form: w = (XᵀX + αI)⁻¹ Xᵀ y — the +αI is what fixes the invertibility problem above.
  • Lasso has no closed form (the absolute value isn't differentiable at 0), so .fit() uses coordinate descent — iteratively optimize one weight at a time, holding the others fixed, until convergence.

Interview trap: "Which gives sparse models?" → Lasso (L1). "Why?" → the L1 penalty's diamond-shaped constraint region has corners on the axes, so the optimum often lands exactly on wⱼ = 0.

3. LogisticRegression — classification learned by gradient-based optimization

Despite the name it's a classifier. It learns weights so that

p(y=1 | x) = σ(Xw + b),   where σ(z) = 1 / (1 + e^−z)   (the sigmoid)

What .fit() minimizes: the log-loss / binary cross-entropy — there is no closed form, so it's solved iteratively by an optimizer (default lbfgs, a quasi-Newton method; others: liblinear, saga).

from sklearn.linear_model import LogisticRegression
clf = LogisticRegression()
clf.fit(X_train, y_train)        # iteratively minimizes log-loss

clf.coef_                        # learned weights (log-odds per feature)
clf.predict(X_test)              # hard class: applies threshold 0.5 to σ(Xw+b)
clf.predict_proba(X_test)        # [[p(class0), p(class1)], ...]
clf.decision_function(X_test)    # raw score Xw+b (before sigmoid)

How it learns, step by step (the answer interviewers want):

  1. Start with weights ≈ 0.
  2. Compute predictions p = σ(Xw + b).
  3. Compute the gradient of log-loss w.r.t. w — conveniently this is Xᵀ(p − y) / n (predicted-minus-actual, like linear regression).
  4. Update weights to reduce the loss (the optimizer decides the step).
  5. Repeat until the loss stops improving (tol) or max_iter is hit.

Interview gold: "Linear vs logistic — same or different learning?" → "Same linear core Xw+b; linear regression has a closed-form least-squares solution, logistic has no closed form and is trained iteratively on cross-entropy because the sigmoid makes it nonlinear in the loss."

4. DecisionTreeClassifier — learning by recursive greedy splitting

No weights at all. .fit() grows a tree by repeatedly asking "which single feature threshold best separates the classes right now?"

How .fit() learns:

  1. At the current node, for every feature and every candidate threshold, compute how much a split would reduce impurity (Gini by default, or entropy).
    • Gini impurity of a node: G = 1 − Σ pᵢ² (pᵢ = fraction of class i).
  2. Pick the single split with the largest impurity decrease (greedy — it never reconsiders).
  3. Partition the data, recurse into each child.
  4. Stop when a stopping rule fires: max_depth, min_samples_split, min_samples_leaf, or the node is pure.
from sklearn.tree import DecisionTreeClassifier
tree = DecisionTreeClassifier(max_depth=3, criterion="gini")
tree.fit(X_train, y_train)

tree.feature_importances_   # learned: total impurity reduction per feature
tree.tree_.node_count       # how many nodes it grew
tree.predict(X_test)        # routes each row down the tree to a leaf's majority class

Interview trap: "Do you need to scale features for a tree?" → No. Trees split on thresholds per-feature, so monotonic rescaling doesn't change splits. (Contrast: linear/SVM/KNN/NN do need scaling.)

5. RandomForestClassifier — bagging many decorrelated trees

.fit() trains N independent trees (n_estimators), each on:

  • a bootstrap sample (random rows, sampled with replacement), and
  • at each split, only a random subset of features (max_features).

These two randomizations decorrelate the trees so their errors cancel when averaged. Prediction = majority vote (classification) or mean (regression).

from sklearn.ensemble import RandomForestClassifier
rf = RandomForestClassifier(n_estimators=300, max_features="sqrt", n_jobs=-1)
rf.fit(X_train, y_train)        # trains 300 trees in parallel (n_jobs=-1)
rf.feature_importances_         # averaged over all trees
rf.predict(X_test)              # majority vote across 300 trees

Interview gold: "Why is a random forest better than one deep tree?" → "A single deep tree has low bias, high variance (overfits). Averaging many decorrelated trees keeps the low bias but slashes variance — that's the bagging effect. The feature subsampling is what decorrelates them; without it all trees would split on the same dominant feature first."

6. GradientBoosting / XGBoost — learning by fitting residuals sequentially

The opposite philosophy to a random forest. Trees are built sequentially, each one correcting the errors of the ensemble so far.

How .fit() learns:

  1. Start with a constant prediction (e.g. the mean).
  2. Compute the residuals / negative gradient of the loss (where the current model is wrong).
  3. Fit a small tree to those residuals.
  4. Add it to the ensemble, scaled by the learning rate (eta).
  5. Repeat for n_estimators rounds.
from sklearn.ensemble import GradientBoostingClassifier
gb = GradientBoostingClassifier(n_estimators=200, learning_rate=0.05, max_depth=3)
gb.fit(X_train, y_train)        # builds 200 trees, each fixing the prior errors

Interview trap: "Bagging vs boosting?" → "Bagging (random forest) trains trees in parallel and independently to reduce variance. Boosting trains trees sequentially, each fitting the previous residuals, to reduce bias. Boosting usually wins on tabular data but is more prone to overfit and needs learning_rate × n_estimators tuning."

7. SVC (Support Vector Machine) — learning the max-margin boundary

.fit() solves a constrained quadratic optimization problem: find the hyperplane that separates classes with the largest margin. Only the points on the margin — the support vectors — matter.

  • The kernel trick (kernel="rbf") lets it learn nonlinear boundaries by implicitly mapping to a higher-dimensional space without computing it.
  • C = regularization (low C → wider margin, more tolerance for misclassification).
from sklearn.svm import SVC
svm = SVC(kernel="rbf", C=1.0, gamma="scale", probability=True)
svm.fit(X_train, y_train)       # solves the quadratic program
svm.support_vectors_            # learned: the points that define the boundary

Gotcha: SVMs are O(n²)–O(n³) in samples → don't scale to millions of rows; and they require feature scaling.

8. KMeans — unsupervised learning by Lloyd's algorithm

No y. .fit(X) learns k cluster centers by alternating two steps until the assignments stop changing (Expectation-Maximization flavor):

  1. Assign each point to its nearest centroid.
  2. Update each centroid to the mean of its assigned points.
  3. Repeat until convergence.
from sklearn.cluster import KMeans
km = KMeans(n_clusters=3, n_init=10, random_state=42)
km.fit(X)
km.cluster_centers_   # learned centroids
km.labels_            # cluster assignment per training point
km.predict(X_new)     # assigns new points to the nearest learned centroid
km.inertia_           # within-cluster sum of squares (lower = tighter)

Interview Qs: "How pick k?" → elbow method on inertia_, or silhouette score. "Why n_init?" → KMeans is sensitive to initialization; run it 10× with different seeds and keep the best inertia_.

9. KNeighborsClassifier — the "lazy" learner that barely fits

.fit() does almost nothing — it just stores the training data (often in a KD-tree/Ball-tree for fast lookup). All the work happens at predict():

from sklearn.neighbors import KNeighborsClassifier
knn = KNeighborsClassifier(n_neighbors=5)
knn.fit(X_train, y_train)   # just memorizes X_train, y_train
knn.predict(X_test)         # for each row: find 5 nearest neighbors, majority vote

Interview gold: "Give an example of a model where fit does no learning." → KNN. It's a lazy/instance-based learner — fit stores data, predict computes distances on the fly. Cost is shifted from training to inference.

10. Putting it together: Pipeline, splitting, and cross-validation

Real interviews want to see you prevent data leakage. The scaler must be fit on train only — a Pipeline guarantees that.

from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import LogisticRegression
from sklearn.model_selection import train_test_split, cross_val_score

X_train, X_test, y_train, y_test = train_test_split(
    X, y, test_size=0.2, stratify=y, random_state=42)

pipe = Pipeline([
    ("scaler", StandardScaler()),          # fit() learns mean_ & scale_ on train
    ("clf", LogisticRegression()),         # fit() learns coef_ on scaled train
])
pipe.fit(X_train, y_train)                 # scaler.fit_transform THEN clf.fit
pipe.score(X_test, y_test)                 # scaler.transform THEN clf.predict

# Cross-validation: refits the whole pipeline on each fold (no leakage)
scores = cross_val_score(pipe, X_train, y_train, cv=5, scoring="f1")
print(scores.mean(), scores.std())

What Pipeline.fit does internally: calls fit_transform on every step except the last, threading the output of one into the next, then fit on the final estimator. At predict time it calls transform on every step (so the scaler uses the train mean_/scale_, never the test set's) then predict on the last.

The #1 leakage trap: scaling/imputing before train_test_split, so test statistics leak into training. The fix is exactly the pipeline above.

  3. Q: Which models need feature scaling? A: Distance/gradient-based — linear, logistic, SVM, KNN, KMeans, neural nets. Not tree-based.
  4. Q: Closed-form vs iterative learners here? A: Closed-form: LinearRegression, Ridge. Iterative: Lasso (coordinate descent), LogisticRegression (lbfgs/saga), SVM (QP), GradientBoosting, KMeans, neural nets.

  7. Q: Why random_state? A: Reproducibility — bootstraps, splits, and KMeans init are stochastic; fixing the seed makes runs deterministic.

Chapter 02b (Phase 2) — Deep Learning with PyTorch

The scikit-learn chapter answered "how does the model learn?" for closed-form and classical iterative learners. Deep learning learns the same way conceptually — minimize a loss by gradient descent — but you write the loop yourself, and the gradients come from autograd / backpropagation. This chapter is the PyTorch equivalent of the .fit() deep dive: tensors, the computational graph, .backward(), the canonical training loop, and the architectures.

1. Tensors — NumPy arrays that live on the GPU and remember their history

import torch

x = torch.tensor([[1., 2.], [3., 4.]])   # 2x2 float tensor
x.shape            # torch.Size([2, 2])
x.dtype            # torch.float32
x @ x              # matrix multiply
x.to("cuda")       # move to GPU  <- the reason PyTorch exists
x.requires_grad_(True)   # start tracking operations for autograd

Two things make a tensor more than a NumPy array:

  1. Device — it can live on a GPU (.to("cuda")), where matmuls run 10–100× faster.
  2. requires_grad — if set, PyTorch records every operation into a computational graph so it can compute gradients later.

Q: tensor vs ndarray? → tensors add GPU execution + autograd. x.numpy() and torch.from_numpy() convert (sharing memory on CPU).

2. Autograd — how gradients are computed (the heart of "learning")

When requires_grad=True, every operation builds a graph of how the output was produced. Calling .backward() walks that graph in reverse, applying the chain rule to fill each tensor's .grad.

w = torch.tensor(3.0, requires_grad=True)
x = torch.tensor(2.0)
y = w * x          # y = 6 ; graph records "y came from w*x"
loss = (y - 10) ** 2   # loss = (6-10)^2 = 16

loss.backward()    # compute d(loss)/d(w) via chain rule
w.grad             # tensor(-16.)  =  2*(y-10)*x = 2*(-4)*2

What .backward() does step by step (this is THE interview answer):

  1. Seeds the gradient of the scalar output with 1.0.
  2. Traverses the graph from output → inputs in reverse topological order.
  3. At each node applies the local derivative times the incoming gradient (chain rule) — this is backpropagation.
  4. Accumulates the result into each leaf tensor's .grad.

Critical gotcha: gradients accumulate (add) across .backward() calls. That's why every training step calls optimizer.zero_grad() first — forget it and gradients from previous batches pile up, corrupting training.

3. The canonical training loop — memorize this cold

Interviewers ask you to write this from memory. The five steps, in order:

import torch, torch.nn as nn

model = nn.Sequential(nn.Linear(10, 32), nn.ReLU(), nn.Linear(32, 1))
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
loss_fn = nn.MSELoss()

for epoch in range(num_epochs):
    for xb, yb in dataloader:                 # mini-batches
        optimizer.zero_grad()                 # 1. clear old gradients
        preds = model(xb)                     # 2. FORWARD pass
        loss = loss_fn(preds, yb)             # 3. compute loss
        loss.backward()                       # 4. BACKWARD: fill .grad
        optimizer.step()                      # 5. update weights: w -= lr*grad
Step Call What happens
1 zero_grad() reset .grad to 0 (else they accumulate)
2 model(xb) runs forward(), builds the graph
3 loss_fn(...) scalar loss = how wrong we are
4 loss.backward() backprop fills every param's .grad
5 optimizer.step() applies the update rule using .grad

Q: what if you swap steps 4 and 5? → step() before backward() updates with stale/empty grads — the model won't learn. Q: forget zero_grad? → gradients accumulate across batches → effectively wrong, exploding updates.

4. nn.Module — defining a model with a custom forward

import torch.nn as nn, torch.nn.functional as F

class MLP(nn.Module):
    def __init__(self, in_dim, hidden, out_dim):
        super().__init__()
        self.fc1 = nn.Linear(in_dim, hidden)   # registers weight + bias params
        self.fc2 = nn.Linear(hidden, out_dim)
    def forward(self, x):                       # YOU define the compute graph
        x = F.relu(self.fc1(x))
        return self.fc2(x)

model = MLP(784, 256, 10)
list(model.parameters())   # all learnable tensors (what the optimizer updates)
  • Assigning nn.Linear/nn.Conv2d as attributes auto-registers their parameters so model.parameters() finds them and optimizer updates them.
  • You never call forward() directly — call model(x), which triggers hooks and then forward.

Q: model.train() vs model.eval()? → toggles training-only layers: Dropout is active in train, off in eval; BatchNorm uses batch stats in train, running averages in eval. Forgetting model.eval() at inference is a classic bug (noisy/incorrect predictions).

5. Optimizers & learning rate — how the update actually happens

# SGD: w = w - lr * grad
torch.optim.SGD(model.parameters(), lr=0.01, momentum=0.9)
# Adam: per-parameter adaptive LR using running averages of grad & grad²
torch.optim.Adam(model.parameters(), lr=1e-3)
  • SGD + momentum: accumulates a velocity to smooth noisy gradients.
  • Adam: adapts the step size per parameter (1st + 2nd moment estimates) — the default for most deep nets, fast convergence, less LR tuning.
  • Learning rate is the most important hyperparameter: too high → diverges (loss → NaN); too low → crawls / stuck.

Q: loss went to NaN — why? → usually LR too high (exploding gradients), or log(0)/divide-by-zero in the loss, or bad input scaling. Fixes: lower LR, gradient clipping (clip_grad_norm_), normalize inputs.

6. The architectures (one-liner each + when)

Arch Core op Used for
MLP / Dense Linear tabular, final heads
CNN Conv2d (weight sharing over space) images, spatial data
RNN/LSTM/GRU recurrent state over time sequences (pre-Transformer)
Transformer self-attention text, and now everything

Self-attention in one breath (you WILL be asked): each token computes a Query, Key, and Value. Attention weight = softmax of Q·Kᵀ / √d_k; the output is the weighted sum of Values. This lets every token "attend to" every other token in parallel — the reason Transformers replaced RNNs (no sequential bottleneck, captures long-range dependencies).

Attention(Q, K, V) = softmax( Q Kᵀ / √d_k ) V

Q: why Transformers over RNNs? → RNNs process tokens sequentially (slow, hard to parallelize, vanishing gradients over long sequences). Attention sees the whole sequence at once, parallelizes on GPUs, and models long-range deps. Q: why divide by √d_k? → keeps the dot products from growing large and pushing softmax into tiny-gradient regions.

7. Regularization & the bias–variance fight in deep nets

  • Dropout — randomly zero activations during training → prevents co-adaptation.
  • Weight decay (L2) — penalizes large weights (set in the optimizer).
  • Batch/Layer Norm — normalize activations → faster, more stable training.
  • Early stopping — stop when validation loss stops improving.
  • Data augmentation — more effective "data" without new labels.

Q: overfitting signs + fixes? → train loss ↓ while val loss ↑. Fixes: more data/augmentation, dropout, weight decay, smaller model, early stopping.

8. Interview Q&A (rapid fire)

  1. Q: backprop in one sentence? → reverse-mode autodiff applying the chain rule from loss to every parameter to get gradients.
  2. Q: vanishing/exploding gradients? → gradients shrink/grow multiplicatively through deep layers; fixes: ReLU, residual connections, normalization, gradient clipping, careful init.
  3. Q: why ReLU over sigmoid in hidden layers? → no saturation for positive inputs → healthier gradients; cheap. Sigmoid saturates → vanishing gradients.
  4. Q: batch vs mini-batch vs SGD? → full-batch (stable, slow/memory-heavy), mini-batch (the practical default), single-sample SGD (noisy).
  5. Q: epoch vs batch vs iteration? → epoch = one full pass over data; batch = subset per step; iteration = one weight update.
  6. Q: with torch.no_grad()? → disables graph building → faster, less memory at inference/eval.
  7. Q: how move training to GPU? → model.to("cuda") and move each batch xb.to("cuda"); both must be on the same device.

  • ML Engineer: write the training loop from memory, explain backprop, debug NaN losses and overfitting, know the architectures. Core round.
  • AI Engineer: understand the Transformer + attention deeply (it's the substrate of every LLM) and inference-time concerns (no_grad, eval, batching); less need to hand-derive backprop.
  • MLOps Engineer: know enough to containerize training, pin CUDA/torch versions, checkpoint/resume, and reason about GPU memory & mixed precision — not to derive gradients.

Chapter 03 (Phase 3) — LLM Engineering & RAG

This is the core AI Engineer chapter. The Transformer chapter (02b) covered how an LLM works internally; this covers how you build with one: tokens, sampling, embeddings, the full Retrieval-Augmented Generation pipeline, and the hardest part of the job — evaluating non-deterministic output. Same code-level depth as the scikit-learn chapter.

1. Tokens & context window — the unit of everything

LLMs don't see characters or words; they see tokens (sub-word pieces). "un- believable" might be 3 tokens. This drives cost, latency, and limits.

import tiktoken
enc = tiktoken.get_encoding("cl100k_base")
enc.encode("RAG is retrieval-augmented generation")   # -> list of token ids
len(enc.encode("hello world"))                         # 2 tokens
# rule of thumb: ~4 chars ≈ 1 token ≈ 0.75 words (English)
  • Context window = max tokens (prompt + completion) the model can attend to (e.g. 8k, 128k, 1M). Exceed it → truncation or API error.
  • You pay per token, input and output separately. Cost optimization = fewer tokens (concise prompts, retrieval instead of stuffing whole docs).

Q: why does the same prompt cost different amounts across models? → different tokenizers + per-token pricing. Q: "context window" vs "training data"? → context = what it sees at inference now; training data = what shaped its weights (frozen).

2. Sampling — temperature, top-p, and why output is non-deterministic

At each step the model outputs a probability distribution over the next token. How you sample from it controls randomness:

Knob Effect
temperature=0 greedy / argmax → (near) deterministic, repetitive
temperature=0.7 balanced creativity (default for chat)
temperature>1 wilder, more diverse, more errors
top_p=0.9 (nucleus) sample only from the smallest set of tokens summing to 90% prob
top_k=40 sample only from the top-k tokens 
client.chat.completions.create(
    model="gpt-4o", messages=msgs,
    temperature=0,          # for extraction/classification → reproducible
    max_tokens=500,
    seed=42,                # best-effort determinism (not guaranteed)
)

Q: how get reproducible output? → temperature=0 (+ seed where supported) — but still not 100% guaranteed (floating-point/batching nondeterminism on the provider side). Q: when high temperature? → brainstorming/creative; never for structured extraction or routing.

3. Embeddings & vector similarity — the basis of retrieval

An embedding maps text → a fixed-length vector where semantic similarity = geometric closeness. "dog" and "puppy" land near each other.

from openai import OpenAI
client = OpenAI()
emb = client.embeddings.create(model="text-embedding-3-small", input="hello")
vec = emb.data[0].embedding          # e.g. 1536 floats

import numpy as np
def cosine(a, b):                    # the standard similarity metric
    a, b = np.array(a), np.array(b)
    return a @ b / (np.linalg.norm(a) * np.linalg.norm(b))
  • Cosine similarity (angle between vectors) is the default — direction matters, magnitude doesn't.
  • A vector database (FAISS, pgvector, Pinecone, Weaviate) stores embeddings and does fast approximate nearest-neighbor (ANN) search.

Q: cosine vs Euclidean? → cosine ignores magnitude (good for text length differences). Q: exact vs approximate NN? → ANN (HNSW/IVF) trades a little recall for huge speed at scale — essential beyond ~100k vectors.

4. RAG — the full pipeline, step by step

Why RAG exists: LLMs hallucinate and have a knowledge cutoff. RAG grounds answers in your own documents by retrieving relevant chunks and putting them in the prompt. This is the #1 AI-Engineer system to be able to design end to end.

INGEST (offline):   docs → chunk → embed → store in vector DB
QUERY  (online):    question → embed → retrieve top-k → [rerank] → stuff into
                    prompt → LLM generates grounded answer (with citations)

# ---- INGEST ----
chunks = chunk_document(doc, size=500, overlap=50)        # 1. chunk
vectors = embed_batch(chunks)                              # 2. embed
vector_db.upsert(ids, vectors, metadata=chunks)           # 3. store

# ---- QUERY ----
def rag_answer(question):
    q_vec = embed(question)                                # 4. embed query
    hits = vector_db.query(q_vec, top_k=5)                 # 5. retrieve
    context = "\n\n".join(h.text for h in hits)
    prompt = f"""Answer ONLY from the context. If not present, say you don't know.
Context:\n{context}\n\nQuestion: {question}"""
    return llm(prompt, temperature=0)                      # 6. generate

The decisions interviewers probe:

  • Chunk size & overlap: too big → noisy retrieval, wasted tokens; too small → lost context. Overlap prevents splitting a fact across a boundary. ~200–800 tokens with 10–20% overlap is typical.
  • top_k: more context = better recall but more tokens/cost and more distraction.
  • Reranking: retrieve 20 with cheap vector search, then a cross-encoder reranker reorders to the best 3–5 → big quality win.
  • Metadata filtering: filter by source/date/permissions before/with the vector search.

Q: RAG vs fine-tuning? → RAG injects knowledge/facts (fresh, citable, cheap to update); fine-tuning changes behavior/style/format (Chapter 06). For "answer from our docs," RAG. For "always output this JSON tone," fine-tune. Often: RAG first, fine-tune only if needed.

5. Hallucination mitigation (the question every AI Engineer gets)

  1. Ground with RAG + instruct "answer only from context; if absent, say you don't know."
  2. Citations — require the model to quote/cite the source chunk; makes fabrication visible and checkable.
  3. Lower temperature for factual tasks.
  4. Guardrails / validation — schema-check structured output; verify claims against retrieved text.
  5. LLM-as-judge / self-check — a second pass asks "is this supported by the context?"

Q: model invents a citation — how prevent? → constrain to retrieved chunk IDs, validate every citation exists in the context post-hoc, reject/retry if not.

Chapter 04 (Phase 4) — Agentic AI

An agent is what you get when you put an LLM in a loop with tools: it can decide what to do next, take an action, observe the result, and repeat until the task is done. This is the frontier AI-Engineer chapter. Interviewers want to see that you understand the loop, the failure modes, and the guardrails — not just that you imported a framework. Same code-level depth as the scikit-learn chapter.

1. Concept map — agent = LLM + tools + loop + memory

            ┌──────────────────────────────────────┐
   goal ───▶│  LLM (the "brain": decides next step) │
            └───────┬───────────────────▲───────────┘
                    │ action (tool call) │ observation (tool result)
                    ▼                    │
            ┌──────────────────────────────────────┐
            │  TOOLS: search, code-exec, DB, API…   │
            └──────────────────────────────────────┘
                    ▲ memory (scratchpad + long-term store)

A plain LLM call is one shot. An agent iterates: think → act → observe → think → … until it decides it's finished. The model's tool-calling ability (Chapter 03 §6) is the primitive that makes this possible.

Q: chatbot vs agent? → a chatbot responds; an agent acts — it chooses and executes tools in a loop to accomplish a multi-step goal, using feedback from each step.

2. The ReAct loop — the canonical pattern (write it from scratch)

ReAct = Reasoning + Acting. The model alternates a Thought, an Action (tool call), and reads back an Observation, looping until a Final Answer.

def react_agent(goal, tools, max_steps=8):
    scratchpad = ""
    for step in range(max_steps):                 # the LOOP (bounded!)
        prompt = f"""Goal: {goal}
You can use tools: {list(tools)}.
Respond with either:
  Thought: ...\nAction: tool_name\nActionInput: ...
or:
  Final Answer: ...
{scratchpad}"""
        out = llm(prompt, temperature=0)

        if "Final Answer:" in out:                # termination
            return out.split("Final Answer:")[-1].strip()

        action = parse_action(out)                # which tool + input
        observation = tools[action.name](action.input)   # EXECUTE the tool
        scratchpad += f"\n{out}\nObservation: {observation}\n"  # feed back
    return "Stopped: step budget exhausted."      # guardrail

Step by step: (1) model emits a Thought + Action; (2) you parse and execute the tool; (3) you append the Observation to the scratchpad; (4) loop with the updated context so the model reasons about what it just learned; (5) stop on "Final Answer" or a step cap.

Q: why bound max_steps? → without a cap an agent can loop forever (and burn money). The step budget is the single most important guardrail.

3. Tools — the agent's hands

A tool = a name + description + typed parameters + a function. The description is part of the prompt — bad descriptions are the #1 reason an agent picks the wrong tool.

tools = {
    "search":  {"desc": "Web search. Input: query string.", "fn": web_search},
    "python":  {"desc": "Run Python. Input: code. Returns stdout.", "fn": run_code},
    "sql":     {"desc": "Query the analytics DB. Input: a SELECT.", "fn": run_sql},
}

Provided to the model via function-calling schemas (Chapter 03 §6). The model returns a structured call; you execute it (the model never runs code itself).

Q: how does the agent 'know' which tool to use? → from the tool names + descriptions in the prompt and the model's reasoning. Precise, disambiguated descriptions + few-shot examples steer selection. Q: security? → tools are an attack surface — sandbox code execution, parameterize SQL, allowlist URLs, never give raw shell to an untrusted prompt (prompt injection).

4. Planning — decompose before acting

For complex goals, agents plan first, then execute sub-steps:

  • Plan-and-execute: generate a full step list up front, then run each (cheaper, more predictable than re-planning every step).
  • ReAct (reactive): decide one step at a time based on observations (more adaptive, more tokens).
  • Reflection / self-critique: after an attempt, the agent critiques its own output and retries (improves quality at cost of more calls).
  • Tree/graph search (ToT): explore multiple reasoning branches, pick the best.

Q: plan-and-execute vs ReAct? → planning up front is cheaper and more controllable but brittle if the world changes mid-task; reactive adapts but costs more and can wander. Hybrids (plan, then react within steps) are common.

5. Memory — short-term vs long-term

Type What Implementation
Short-term the current task's scratchpad / conversation the context window (finite!)
Long-term facts across sessions a vector store (RAG over past interactions)
Working/state variables, intermediate results explicit external state (a dict/DB)

When the conversation outgrows the context window you summarize or retrieve: keep a running summary + retrieve relevant past turns by embedding similarity (RAG, Chapter 03).

Q: how does an agent remember beyond the context window? → externalize memory: summarize old turns, and store/retrieve facts from a vector DB keyed by relevance. The context window is RAM; the vector store is disk.

6. Multi-agent systems

Split a hard task across specialized agents coordinated by an orchestrator:

Orchestrator ──▶ Researcher (gathers) ──▶ Coder (implements) ──▶ Critic (reviews)
                         ▲                                            │
                         └────────────── feedback ────────────────────┘

  • Cons: more cost/latency, harder to debug, errors compound across hops.

Q: when multi-agent vs one agent with many tools? → start with one capable agent; reach for multi-agent only when roles are genuinely distinct or you need parallel workstreams. Each hand-off adds error and cost.

Chapter 05 (Phase 5) — AI Infrastructure (Model Serving & GPUs)

This is where ML meets systems: how do you actually serve a model — especially an LLM — fast, cheap, and at scale? It's core for MLOps Engineers and a strong differentiator for AI Engineers. Interviewers probe GPU memory math, the KV-cache, batching, and the throughput-vs-latency tradeoff. Maps directly to the NVIDIA NCA-AIIO cert. Same code-level depth as the scikit-learn chapter.

1. Concept map — training vs inference are different problems

Training Inference (serving)
Runs once (or periodically) constantly, per request
Bottleneck compute (FLOPs), data memory + latency
Batch large, offline small, online, latency-bound
Metric time-to-train, accuracy latency (p50/p99), throughput, $/token

Serving is the 90% of an LLM's lifetime cost. The whole game is getting more tokens/second per GPU dollar without blowing latency.

2. GPU memory math — the question that filters candidates

How much VRAM does a model need just to load?

params × bytes_per_param = weight memory
  FP32 = 4 bytes,  FP16/BF16 = 2 bytes,  INT8 = 1 byte,  INT4 = 0.5 byte
  • A 7B model in FP16: 7e9 × 2 = 14 GB just for weights.
  • A 70B model in FP16: 70e9 × 2 = 140 GB → won't fit on one 80 GB A100/H100 → needs quantization or multi-GPU sharding.
  • Inference also needs the KV-cache (§4) and activation memory on top — budget ~1.2× weights as a rough floor.
training memory ≈ weights + gradients + optimizer state
  Adam FP32: ~ params × (4 + 4 + 8) = 16 bytes/param  → 7B needs ~112 GB to train

Q: can a 13B model run on a 24 GB GPU? → FP16 is 26 GB → no, not as-is; quantize to INT8 (13 GB) or INT4 (6.5 GB) and it fits with room for the KV-cache. This back-of-envelope is the single most-asked infra question.

3. Quantization — trading precision for memory/speed

Represent weights/activations in fewer bits:

Precision Bytes/param Use
FP32 4 training reference
FP16/BF16 2 standard serving (BF16 = wider range, training-friendly)
INT8 1 ~2× smaller, minor quality loss (GPTQ/AWQ)
INT4 0.5 ~4× smaller, fits big models on small GPUs, more loss

Q: does quantization hurt accuracy? → some, but modern methods (GPTQ, AWQ, SmoothQuant) keep it small by quantizing smartly (per-channel, preserving outliers). INT8 is usually "free"; INT4 is a real but often acceptable trade. Q: FP16 vs BF16? → same size; BF16 has a wider exponent range (fewer overflow/NaN issues), preferred for training and modern serving.

4. The KV-cache — why LLM inference is memory-bound

When generating token by token, attention needs the Keys and Values of all previous tokens. Recomputing them every step is O(n²); instead they're cached.

  • The KV-cache grows with sequence length × batch size and can dwarf the model weights for long contexts.
  • KV-cache size ≈ 2 (K&V) × layers × seq_len × batch × hidden × bytes.
  • This is why long context is expensive and why memory, not FLOPs, caps LLM serving throughput.

Q: why is LLM generation memory-bandwidth-bound, not compute-bound? → each new token reads the entire KV-cache + weights from GPU memory but does little compute; you're limited by how fast you can move memory, not by FLOPs. Q: what is PagedAttention (vLLM)? → manages the KV-cache like OS virtual memory (pages), eliminating fragmentation so you fit more concurrent sequences per GPU.

5. Batching — the throughput lever

GPUs are massively parallel; processing one request wastes them. Batch many requests together.

  • Static batching: wait, collect N requests, run together — simple but adds latency and stalls on the slowest/longest sequence.
  • Continuous (in-flight) batching: the serving engine swaps finished sequences out and new ones in every step, keeping the GPU saturated. This is the big vLLM/TGI win — huge throughput gains at similar latency.

Q: throughput vs latency tradeoff? → bigger batches = higher throughput (tokens/sec across users) but higher per-request latency. You tune batch size / max-num-seqs to hit a p99 latency SLA while maximizing throughput.

Q: how serve a 70B model that doesn't fit on one GPU? → tensor parallelism (--tensor-parallel-size N) shards each layer across N GPUs; pipeline parallelism splits layers across GPUs/nodes. Plus quantization.

7. Scaling & cost

  • Horizontal scaling: more replicas behind a load balancer (Chapter 07 — K8s
    • an autoscaler). LLM autoscaling keys off GPU utilization / queue depth, not CPU.
  • Cold starts: loading a big model takes time → keep warm pods, or accept scale-to-zero latency for cheap/bursty workloads.
  • Cost levers: quantization, batching, smaller/distilled models, spot GPUs for batch jobs, caching identical requests, routing easy queries to cheaper models.
  • Speculative decoding: a small draft model proposes tokens the big model verifies in parallel → faster generation, same output distribution.

Q: GPU utilization is low but latency is high — why? → likely small batches / poor batching (GPU idle waiting), or memory-bound KV-cache limiting concurrency. Fix with continuous batching and tuning max-num-seqs.

  2. Q: why is inference memory-bound? → token generation moves weights + KV-cache through memory per step with little compute → bandwidth-limited.
  3. Q: latency vs throughput? → latency = one request's time; throughput = requests/tokens per second across all users; batching trades one for the other.
  4. Q: p50 vs p99 latency? → median vs tail; SLAs care about p99 (worst-case user experience), which batching/queueing can hurt.
  5. Q: data vs model vs tensor vs pipeline parallelism? → data = same model, split batch; tensor = split each layer's math across GPUs; pipeline = split layers across GPUs; model parallel = umbrella for tensor+pipeline.
  6. Q: how cut serving cost 50%? → quantize (INT8/4), continuous batching, distill to a smaller model, cache, route by difficulty, spot instances.
  7. Q: GPU vs CPU for inference? → GPUs win on large parallel matmuls (LLMs); small models / low QPS / latency-insensitive batch can be cheaper on CPU.

Chapter 06 (Phase 6) — Fine-Tuning

Fine-tuning means continuing to train a pretrained model on your own data to change its behavior (style, format, a skill, a domain). The interview centers on three things: when to fine-tune vs RAG vs prompt, how LoRA/QLoRA make it cheap, and how alignment (RLHF/DPO) works. Maps to the Generative-AI-with-LLMs and IBM watsonx certs. Same code-level depth as the scikit-learn chapter.

1. The decision tree — fine-tune vs RAG vs prompt (asked first, every time)

Need the model to KNOW new facts?        → RAG  (inject knowledge at query time)
Need the model to BEHAVE differently?    → Fine-tune (style/format/skill/tone)
Just need better instructions/examples?  → Prompt engineering / few-shot
Prompt RAG Fine-tune
Changes nothing (inference only) what's in context the weights
Good for quick behavior nudges fresh, citable knowledge consistent behavior/format/skill
Update cost instant re-embed docs re-train
Risk limited control retrieval quality catastrophic forgetting, cost

Q: "We want answers from our 10k internal docs." Fine-tune or RAG? → RAG. Facts change and need citations; fine-tuning bakes them in statically and hallucination-prone. Q: "We want every reply in our brand voice + strict JSON." → fine-tune (behavior/format), or a tight prompt if simple.

2. Full fine-tuning vs PEFT — why nobody updates all the weights anymore

Full fine-tuning updates every parameter → needs optimizer state for all of them (Chapter 05 §2: ~16 bytes/param for Adam) → a 7B model needs ~100+ GB and you store a full copy per task. Impractical for most teams.

PEFT (Parameter-Efficient Fine-Tuning) freezes the base model and trains a tiny set of new parameters (<1%). Cheap, fast, and you store only the small delta per task.

Q: why not full fine-tune? → memory + cost + you'd keep a full model copy per task, and small datasets cause catastrophic forgetting. PEFT (LoRA) gets ~the same quality, trains on one consumer GPU, and yields swappable adapters.

3. LoRA — how it actually works (the mechanism interviewers want)

Low-Rank Adaptation. Insight: the update to a weight matrix during fine-tuning is low-rank — it can be approximated by two skinny matrices.

For a frozen weight W (d×d), instead of learning ΔW (d×d, huge), learn ΔW = B·A where A is r×d and B is d×r, with rank r ≪ d (e.g. r=8, 16):

h = W·x + (B·A)·x · (α/r)
    └frozen┘  └─ trained ─┘
  • Only A and B train; W stays frozen. For d=4096, r=8: you train 2 × 4096 × 8 ≈ 65k params instead of 16.7M per matrix — ~0.4%.
  • α (lora_alpha) scales the update; r is the rank/capacity.
from peft import LoraConfig, get_peft_model
config = LoraConfig(r=16, lora_alpha=32, lora_dropout=0.05,
                    target_modules=["q_proj", "v_proj"], task_type="CAUSAL_LM")
model = get_peft_model(base_model, config)
model.print_trainable_parameters()   # e.g. "trainable: 0.06% of params"
  • Adapters are swappable: ship one base model + many small LoRA files (a few MB each), hot-swap per task. Optionally merge B·A back into W for zero inference overhead.

Q: what does LoRA freeze and train? → freezes all original weights, trains only the injected low-rank A/B matrices on chosen layers (usually attention projections). Q: what's r? → the rank = adapter capacity; higher r = more expressive but more params; 8–64 typical.

4. QLoRA — LoRA on a quantized base (fine-tune a 70B on one GPU)

QLoRA = quantize the frozen base to 4-bit (NF4), then train LoRA adapters on top in 16-bit. The frozen base barely costs memory; gradients flow only through the small adapters.

from transformers import BitsAndBytesConfig
bnb = BitsAndBytesConfig(load_in_4bit=True, bnb_4bit_quant_type="nf4",
                         bnb_4bit_compute_dtype="bfloat16")
base = AutoModelForCausalLM.from_pretrained(model_id, quantization_config=bnb)
# then attach LoRA as in §3 -> fine-tune a 33B/70B on a single 24-48GB GPU

Q: difference LoRA vs QLoRA? → QLoRA additionally 4-bit quantizes the frozen base (NF4) to slash memory, enabling big-model fine-tuning on one GPU; the adapters still train in higher precision. Tiny quality cost, huge memory win.

Q: signs of bad fine-tune? → overfits/parrots training data, catastrophic forgetting of general ability, or format drift — mitigate with small LR, few epochs, eval on held-out + general benchmarks.

6. Alignment — RLHF and DPO (how chat models are made helpful)

Pretraining gives a next-token predictor; alignment makes it follow instructions and human preferences. Pipeline:

  1. SFT — supervised fine-tune on instruction→response demos.
  2. Reward model — train a model to score which of two responses humans prefer.
  3. RLHF (PPO) — use RL to push the LLM toward high-reward responses, with a KL penalty so it doesn't drift too far from the SFT model.

DPO (Direct Preference Optimization) skips the separate reward model + RL: it optimizes the preference objective directly on (chosen, rejected) pairs — far simpler and more stable, now the common choice.

from trl import DPOTrainer            # data: {prompt, chosen, rejected}

Q: RLHF vs DPO? → both align to human preferences; RLHF trains a reward model then does PPO (powerful, finicky, unstable); DPO optimizes directly on preference pairs (simpler, cheaper, stable) — usually the default now. Q: why the KL penalty in RLHF? → keep the tuned model close to the reference so it doesn't "reward-hack" into degenerate text.

Chapter 07 (Phase 7) — Kubernetes & Cloud

This is the MLOps core chapter and maps directly to two certs: CKA (Kubernetes) and HashiCorp Terraform Associate. 1. Concept map — what Kubernetes actually is

Kubernetes (K8s) is a container orchestrator: you declare the desired state (in YAML), and the control plane continuously reconciles reality toward it (restarting crashed pods, rescheduling on dead nodes, scaling to match).

Control plane:  API server ── etcd (state) ── scheduler ── controller-manager
Worker nodes :  kubelet + container runtime + kube-proxy  → run your Pods

The object hierarchy (know this cold):

Deployment  ──manages──▶ ReplicaSet ──manages──▶ Pods ──contain──▶ Containers
   │                                                  ▲
 Service (stable network endpoint + load-balances) ───┘
 Ingress (HTTP routing from outside) ──▶ Service
  • Pod — smallest unit; one or more containers sharing network/storage. Pods are ephemeral (die and get replaced with new IPs).
  • Deployment — declares N replicas of a pod template; handles rolling updates and self-healing.
  • Service — a stable virtual IP/DNS name that load-balances across the current pods (because pod IPs churn).
  • Ingress — L7 HTTP(S) routing (host/path) into Services.

Q: pod vs deployment vs service? → pod = running container(s) (ephemeral); deployment = keeps N pods alive + rolls out updates; service = stable endpoint load-balancing across them. Q: why not hit a pod directly? → pod IPs change on restart/reschedule; the Service gives a durable address.

      nodeSelector: { gpu-type: a100 }
  • GPUs are requested as an extended resource; the NVIDIA device plugin exposes them. GPUs aren't fractionally shared by default (one pod per GPU) — MIG / time- slicing can partition them.
  • Use taints/tolerations to keep non-GPU pods off expensive GPU nodes, and node affinity to place training on the right hardware.

Q: why taint GPU nodes? → so only workloads that tolerate the taint (your GPU jobs) land there — you don't want a web pod squatting a $30k/mo GPU node.

terraform plan      # SHOW the diff (create/update/destroy) — review before apply

Q: why plan before apply? → preview exactly what will be created/changed/ destroyed — catch an accidental DB drop before it happens. Q: why remote state + locking? → local state doesn't scale to teams; remote + lock prevents concurrent applies from corrupting it. Chapter 08 (Phase 8) — Observability & Monitoring

A model that passed eval offline can silently rot in production as the world shifts under it. Observability is how you know — and it's a favorite MLOps interview topic because it separates "I trained a model" from "I operated one." This chapter covers the three pillars, the ML-specific failure (drift), the detection math, and LLM observability. Same depth as the scikit-learn chapter.

1. Concept map — software observability vs ML observability

Classic apps have three pillars:

Pillar Answers Tool
Metrics "how much / how fast?" (rates, latency, errors) Prometheus + Grafana
Logs "what happened?" (discrete events) Loki / ELK
Traces "where did the time go?" (request across services) OpenTelemetry / Jaeger

ML systems add a fourth concern the others miss: the code is fine, the service is up, latency is green — but predictions are getting worse because the input distribution drifted. You must monitor data + predictions, not just infra.

Q: why isn't standard APM enough for ML? → it tells you the service is healthy, not that the model is still accurate. A drifted model returns 200 OK with confidently wrong answers. You need data/prediction monitoring on top.

2. The four golden signals (infra layer) + Prometheus

The SRE "golden signals": Latency, Traffic, Errors, Saturation.

from prometheus_client import Counter, Histogram
REQS = Counter("predict_requests_total", "count", ["status"])
LAT  = Histogram("predict_latency_seconds", "latency")

@LAT.time()
def predict(x):
    try:
        y = model(x); REQS.labels("ok").inc(); return y
    except Exception:
        REQS.labels("error").inc(); raise

histogram_quantile(0.99, rate(predict_latency_seconds_bucket[5m]))  # p99 latency
rate(predict_requests_total{status="error"}[5m])                    # error rate

Q: why histogram not average latency? → averages hide the tail; you need p95/p99 quantiles because a few slow requests ruin UX and SLAs. Prometheus histograms bucket observations so you can compute quantiles server-side.

3. Drift — the ML-specific failure (the core of this chapter)

Type What changed Example
Data / covariate drift input distribution P(X) a new user demographic, sensor recalibrated
Concept drift the relationship P(Y|X) fraud patterns evolve; same inputs, new meaning
Label/prior drift output distribution P(Y) spam rate jumps seasonally
Training/serving skew train data ≠ prod data pipeline feature computed differently online vs offline
  • Data drift you can detect immediately (inputs only).
  • Concept drift needs ground-truth labels (often delayed) — you only learn the model is wrong once outcomes arrive.

Q: data drift vs concept drift? → data drift = the inputs shifted (P(X)); concept drift = the input→output relationship shifted (P(Y|X)). Data drift may not hurt accuracy; concept drift always does. Q: training/serving skew? → the features at serving time differ from training (different code path, missing normalization, stale feature store) — a top silent killer; fix by sharing the transformation code / feature store between train and serve.

4. Detecting drift — the math interviewers probe

You compare a reference distribution (training/last-good) to the current production window:

  • PSI (Population Stability Index) — bins a feature, sums Σ (cur% − ref%) · ln(cur%/ref%). Rule of thumb: **<0.1 stable, 0.1–0.25 moderate,

    0.25 significant drift**.

  • KS test (Kolmogorov–Smirnov) — for continuous features, max gap between two CDFs; p-value < 0.05 → distributions differ.
  • Chi-square — for categorical features.
  • Embedding/feature-distance — for unstructured data (text/images), compare embedding distributions (e.g. MMD).
from scipy.stats import ks_2samp
stat, p = ks_2samp(reference_feature, production_feature)
if p < 0.05:
    alert("data drift detected on feature X")   # investigate / retrain

Q: how detect drift without labels? → statistical tests on the input features (PSI/KS) and on the prediction distribution (are we suddenly predicting 'fraud' 3× more?). Labels aren't needed for data drift — only for confirming accuracy loss. Q: PSI threshold? → >0.25 is a strong drift signal; 0.1–0.25 warrants investigation.

5. The ML monitoring stack & retraining loop

predict ──▶ log (input features + prediction + model version) ──▶ store
                                   │
   drift checks (PSI/KS) ◀─────────┤
   delayed ground truth ───────────┤──▶ accuracy/precision over time
                                   ▼
            ALERT → investigate → retrain → validate → canary → promote
  • Log every prediction with input features, output, confidence, model version, and a request id (so you can join to ground truth later).
  • Shadow / canary new models against live traffic before full promotion.
  • Automated retraining triggers: scheduled, drift-threshold-based, or performance-drop-based — always validate + canary before promote (never blind auto-deploy).

Q: when retrain? → on a schedule, on drift past threshold, or on measured accuracy drop — but gate every retrain behind eval + canary so a bad retrain can't ship. Q: how attribute a prod accuracy drop? → join logged predictions to delayed labels, slice by segment/feature, check drift metrics + skew, and compare model versions.

6. LLM observability (the new, hot sub-topic)

LLM apps need everything above plus generation-specific signals:

  • Cost & tokens — per request input/output tokens × price; the #1 runaway cost.
  • Latency — time-to-first-token (streaming UX) and total.
  • Quality/safety — hallucination rate, groundedness, toxicity, refusal rate, often via online LLM-as-judge sampling (Chapter 03 §7).
  • Tracing — for RAG/agents, trace the full chain (retrieval hits → prompt → tool calls → output) to debug why an answer was bad.
  • Tools: LangSmith, Langfuse, Arize/Phoenix, OpenTelemetry GenAI conventions.

Q: how monitor a RAG app in prod? → trace each step (retrieved chunks, final prompt, tokens, latency, cost), sample outputs for an LLM-judge faithfulness score, track retrieval hit quality, and alert on cost/latency/ low-groundedness spikes.

7. Alerting — make it actionable, not noisy

  • Alert on symptoms users feel (error rate, p99 latency, accuracy drop, cost spike), not every metric.
  • SLO/SLI + error budgets: define a target (99.9% of requests < 300 ms); alert when you're burning the budget too fast.
  • Avoid alert fatigue: severity tiers, dedup, sensible thresholds; every page should be actionable.

Q: SLI vs SLO vs SLA? → SLI = the measured metric; SLO = your internal target for it; SLA = the contractual promise (with penalties). Q: how avoid alert fatigue? → alert on user-facing symptoms + error budgets, not raw resource noise; tune thresholds; group/dedup.

8. Interview Q&A (rapid fire)

  1. Q: three pillars of observability? → metrics, logs, traces (+ ML: data/ prediction monitoring).
  2. Q: monitoring vs observability? → monitoring = watching known metrics; observability = being able to ask new questions about why from rich telemetry.
  3. Q: most dangerous ML prod failure? → silent drift / training-serving skew — green dashboards, wrong predictions.
  4. Q: detect drift without labels? → statistical tests (PSI/KS/chi-square) on inputs + prediction distribution shift.
  5. Q: what to log per prediction? → features, output, confidence, model version, request id, timestamp (to join to delayed labels).
  6. Q: p50 vs p99 — which for SLA? → p99 (tail) — it's the user pain you promise against.
  7. Q: shadow vs canary? → shadow = run new model on copied traffic, don't serve its output (compare); canary = serve to a small % and watch metrics before ramp.

Chapter 09 (Phase 9) — Production AI Systems & System Design

The ML/LLM system-design round is the highest-signal interview and the one candidates most often fluff. It's open-ended ("design a recommendation system", "design a customer-support RAG assistant") and tests whether you can connect every prior chapter into one coherent, scalable, monitored system. This chapter gives you a repeatable framework + two fully worked examples.

1. The framework — drive the conversation in this order

Don't jump to "I'd use a transformer." Walk the interviewer through a structure. Spend the first 5 minutes on requirements — the #1 differentiator.

1. CLARIFY    scope, users, scale, latency budget, success metric
2. FRAME      ML problem type? (classif / rank / regress / generate / retrieve)
3. DATA       sources, labels, volume, freshness, privacy
4. FEATURES   feature engineering + feature store (online/offline parity)
5. MODEL      baseline → candidate models, why; offline metric
6. EVAL       offline metric + ONLINE metric (A/B), guardrails
7. SERVE      batch vs real-time, latency, infra (Ch 5/7), API
8. SCALE      QPS, caching, sharding, cost
9. MONITOR    drift, skew, retraining loop (Ch 8)
10. ITERATE   failure modes, next improvements

Q: where do candidates lose this round? → skipping requirements (building the wrong thing), no online vs offline metric distinction, and forgetting the production half (serving/scale/monitoring). Narrate tradeoffs out loud.

2. The vocabulary of tradeoffs (sprinkle these, with justification)

  • Latency vs accuracy — a bigger model is better but slower; distill/quantize or use a two-stage (cheap recall → expensive rerank).
  • Batch vs real-time — precompute offline (cheap, stale) vs compute per-request (fresh, costly). Recommendations are often hybrid.
  • Build vs buy — managed API (fast, opex, less control) vs self-host (capex, control, data privacy).
  • Throughput vs latency — batching (Ch 5).
  • Freshness vs cost — how often to retrain/re-embed.
  • Precision vs recall — pick per business cost of FP vs FN (fraud: recall; spam: precision-leaning).

3. The two-stage pattern (recall → rank) — know this cold

Almost every large-scale ranking/recsys/retrieval system is two-stage:

millions of items
      │  STAGE 1: CANDIDATE GENERATION (recall)
      ▼  cheap, fast, high-recall — embeddings + ANN, get top ~1000
   ~1000 candidates
      │  STAGE 2: RANKING (precision)
      ▼  expensive model scores each, returns top ~10
   final results

Why: you can't run a heavy model over millions of items per request. Cheap recall narrows the field; an expensive ranker perfects the top. (This is exactly RAG's retrieve→rerank, Chapter 03 §4.)

4. Worked example A — "Design a video recommendation system"

  1. Clarify: homepage feed; 100M users, 10M videos; <200 ms; metric = watch time + engagement; cold-start matters.
  2. Frame: ranking/recommendation → personalized scoring of candidates.
  3. Data: watch history, likes, dwell time, video metadata/embeddings; implicit feedback (watches) >> explicit (ratings).
  4. Features: user embedding, item embedding, context (time/device), cross features; feature store for online/offline parity (avoid skew, Ch 8).
  5. Model: Stage 1 — two-tower (user-tower & item-tower) → ANN retrieval of ~1k candidates. Stage 2 — gradient-boosted trees or a DNN ranker on rich features.
  6. Eval: offline = AUC/NDCG; online = A/B test on watch time; guardrails on diversity/clickbait.
  7. Serve: precompute item embeddings nightly (batch); user tower + ANN + ranker real-time. Cache per-user candidates.
  8. Scale: ANN index sharded; ranker behind autoscaled service (Ch 7).
  9. Monitor: feature drift, CTR drop, training/serving skew; retrain cadence.
  10. Cold start: popularity/exploration for new users/items; bandits.

Q: implicit vs explicit feedback? → implicit (clicks/watches) is abundant but noisy/biased (position bias); explicit (ratings) is sparse but clean. Most systems lean implicit + debias.

5. Worked example B — "Design a customer-support RAG assistant"

  1. Clarify: answer from 50k help-center docs + tickets; cite sources; <3 s; metric = resolution rate + CSAT; must say "I don't know" / escalate.
  2. Frame: RAG (Ch 3), optionally agentic for actions (refunds → tools, Ch 4).
  3. Data: docs, past resolved tickets; chunk + embed (Ch 3 §4).
  4. Pipeline: query → embed → vector retrieve top-k → rerank → prompt with context + citation instruction → LLM (temp 0) → guardrail check.
  5. Model choice: strong hosted LLM for quality, or self-host for privacy/cost; smaller model for routing.
  6. Eval: golden Q→A set, faithfulness/groundedness (LLM-judge, Ch 3 §7), online resolution rate + escalation rate; red-team for prompt injection.
  7. Serve: stateless API; vector DB (pgvector/Pinecone); stream tokens for UX.
  8. Scale: cache frequent Q&A, batch embeddings, route easy→cheap model.
  9. Monitor: cost/tokens, latency, groundedness, hallucination/escalation rates, retrieval hit quality (Ch 8 §6).
  10. Safety: "answer only from context," human handoff for low-confidence / sensitive actions, PII handling.

Q: how stop it answering outside the docs? → strict grounding prompt + retrieval-confidence threshold + post-hoc citation validation + fallback to "let me connect you to an agent." Q: how measure success? → online resolution rate & CSAT, not just offline answer similarity.

6. Reliability & safety cross-cuts (mention them)

  • Graceful degradation: model/vector-DB down → fallback (cached answer, rules, human). Never hard-fail the user.
  • Idempotency & retries for agent actions; human-in-the-loop for irreversible ops (Ch 4 §7).
  • Privacy/compliance: PII redaction, data residency, audit logs.
  • Cost guardrails: token/$$ budgets, rate limits, caching.

7. Interview Q&A (rapid fire)

  1. Q: first thing in a system-design round? → clarify requirements + scale + success metric before any architecture.
  2. Q: offline vs online evaluation? → offline = held-out metrics pre-launch; online = A/B test on the real business metric. Offline ≠ online; always A/B.
  3. Q: why two-stage retrieval/ranking? → can't run a heavy model over millions of items; cheap recall narrows, expensive ranker perfects the top.
  4. Q: batch vs real-time inference — pick one when? → batch for non-urgent / precomputable (nightly recs, embeddings); real-time for per-request freshness.
  5. Q: how A/B test an ML model? → split traffic, hold everything else constant, measure the business metric with significance; watch guardrail metrics; ramp.
  6. Q: handle cold start? → popularity baselines, content features, exploration/bandits until you have interaction data.
  7. Q: how keep it from failing hard? → fallbacks, caching, circuit breakers, timeouts, and a non-ML default path.

  • MLOps Engineer: lead serving/scale/monitoring/retraining; the production half is your spotlight.

Chapter 10 (Phase 10) — Data Center & Rack-Scale Systems (Stretch)

The stretch chapter: how very large models are trained across thousands of GPUs. Most ML/AI/MLOps roles won't drill this deeply, but for AI-infrastructure roles (NVIDIA NCA-AIIO territory) and senior/staff interviews, understanding why distributed training is hard — interconnects, parallelism strategies, and the all-reduce — is a real differentiator. Same code-level depth as the scikit-learn chapter.

1. Concept map — why one GPU is never enough for frontier models

A frontier model (100B–1T+ params) cannot fit on one GPU (Chapter 05 §2: a 70B model in FP16 = 140 GB > 80 GB H100), and even if it did, training on one device would take years. So training is distributed across many GPUs, many nodes, many racks — and the bottleneck shifts from compute to communication.

GPU ── NVLink ──▶ GPU        (intra-node: ~900 GB/s, very fast)
 node ── InfiniBand ──▶ node  (inter-node: ~400 Gb/s, slower)
 rack ── spine/leaf network ──▶ rack

Q: why is distributed training hard? → it's a communication problem. GPUs are fast; keeping thousands of them in sync (exchanging gradients) without starving them is the challenge — the interconnect, not the math, is the limit.

2. The interconnect hierarchy — bandwidth is everything

Link Scope Rough BW Note
NVLink / NVSwitch GPU↔GPU in a node ~900 GB/s why 8-GPU nodes are a unit
PCIe GPU↔CPU/host ~64 GB/s much slower than NVLink
InfiniBand / RoCE node↔node ~400 Gb/s RDMA, low latency
Spine-leaf fabric rack↔rack — non-blocking topologies

The whole game: keep communication local (NVLink within a node) and minimize the slow inter-node traffic. Parallelism strategies are chosen around this.

Q: NVLink vs PCIe vs InfiniBand? → NVLink = ultra-fast GPU-to-GPU inside a server; PCIe = host↔GPU (slower); InfiniBand = fast low-latency server-to-server with RDMA. Q: what is RDMA? → Remote Direct Memory Access — one node reads/ writes another's memory bypassing the CPU/OS, cutting latency for gradient sync.

3. The parallelism strategies (the core taxonomy)

You combine these ("3D parallelism") to train huge models:

Strategy Splits Communication
Data parallel (DP) the batch (each GPU gets a full model copy, different data) all-reduce gradients each step
Tensor parallel (TP) each layer's matrices across GPUs heavy per-layer comms → keep within a node (NVLink)
Pipeline parallel (PP) layers across GPUs/nodes (stages) activations between stages; needs micro-batching to avoid "bubbles"
ZeRO / FSDP optimizer state, gradients, params sharded across DP ranks gathers shards on demand → DP without N full copies 
# PyTorch FSDP (Fully Sharded Data Parallel) — shard a big model across GPUs
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
model = FSDP(model)        # each rank holds a shard; gathers params per layer

Q: data vs model parallelism? → data parallel = same model copied, split the batch (simplest, comms = gradient all-reduce); model parallel = split the model itself (tensor = within layers, pipeline = across layers) when it doesn't fit. Q: what does ZeRO solve? → plain data-parallel keeps a full copy of params+grads+optimizer state on every GPU (wasteful); ZeRO shards them across ranks, cutting per-GPU memory ~N× so you can data-parallel huge models.

4. The all-reduce — the operation at the heart of data-parallel training

In data-parallel training, after each backward pass every GPU has its own gradients from its slice of the batch. They must be averaged across all GPUs before the optimizer step — that's the all-reduce collective.

Step: forward → backward (local grads) → ALL-REDUCE (sum/avg grads across GPUs)
      → every GPU now has identical averaged grads → optimizer.step (in sync)
  • Ring all-reduce (NCCL): GPUs form a ring; gradients are chunked and passed around so bandwidth use is optimal and independent of GPU count.
  • This sync happens every step — if the interconnect is slow, GPUs sit idle waiting → the dominant scaling bottleneck.

Q: what is all-reduce and why does it matter? → it sums/averages each GPU's gradients so all replicas stay identical; it's the per-step sync that makes data-parallel work, and its cost (interconnect-bound) is what limits scaling efficiency. Q: what is NCCL? → NVIDIA Collective Communications Library — the optimized all-reduce/all-gather/broadcast primitives over NVLink/InfiniBand.

5. Scaling efficiency, failures, and the physical layer

  • Scaling efficiency: doubling GPUs rarely doubles speed — comms overhead and pipeline "bubbles" cause sub-linear scaling. Measured as % of ideal linear speedup (MFU — Model FLOPs Utilization — is the standard metric).
  • Fault tolerance: at 10k GPUs, hardware fails during a run routinely → checkpoint frequently so you resume, not restart. A multi-week run that can't checkpoint is a non-starter.
  • Power & cooling: a rack of H100s draws tens of kW; data centers are increasingly power-limited and moving to liquid cooling. PUE (Power Usage Effectiveness) measures DC efficiency.
  • Topology-aware placement: the scheduler places tightly-communicating ranks on the same node/rack to keep traffic on fast links.

Q: why checkpoint during training? → at scale, a GPU/node will fail mid-run; checkpoints let you resume from the last good state instead of losing weeks. Q: why is scaling sub-linear? → communication overhead (all-reduce), pipeline bubbles, and stragglers; you measure MFU, not raw GPU count.

6. Interview Q&A (rapid fire)

  1. Q: why distribute training at all? → models + data don't fit/finish on one GPU; you need many for memory and throughput.
  2. Q: the three parallelism axes? → data (split batch), tensor (split layer math), pipeline (split layers); combined = 3D parallelism.
  3. Q: what travels over the network in data-parallel? → gradients, via all-reduce, every step.
  4. Q: why keep tensor-parallel within a node? → it has heavy per-layer comms; NVLink (intra-node) is far faster than InfiniBand (inter-node).
  5. Q: ZeRO/FSDP in one line? → shard params/grads/optimizer state across data- parallel GPUs to cut per-GPU memory.
  6. Q: biggest non-compute constraint in modern DCs? → power & cooling (and the interconnect); frontier training is increasingly power-bound.
  7. Q: gradient accumulation — why? → simulate a large batch on limited memory by summing grads over several micro-batches before one optimizer step.

  • MLOps Engineer: know enough to run multi-GPU/multi-node training jobs (torchrun, FSDP, checkpointing) and provision GPU clusters (Ch 7).
  • ML Engineer: understand data vs model parallelism and gradient accumulation to scale your own training; rarely asked the rack-level physics.
  • AI Engineer: least relevant — you consume trained models; helpful context for cost/feasibility conversations.

  • Saying "just add more GPUs" without acknowledging comms overhead / sub-linear scaling (§5).
  • Confusing tensor vs pipeline vs data parallelism (§3).
  • Not knowing all-reduce is the per-step gradient sync (§4).
  • Ignoring fault tolerance / checkpointing at scale (§5).

Chapter 12 — Behavioral & Role-Specific Scenarios

Technical chops get you to the onsite; behavioral rounds decide the offer and the level. 1. STAR — the structure that makes stories land

Every behavioral answer should follow STAR, ~2 minutes, mostly the A:

Letter What Keep it

T — Task your specific responsibility 1 sentence — your part, not "we"

Q: how detailed should a STAR answer be? → 90 seconds–2 minutes; spend ~60% on Action. The #1 fix: say "I" not "we" — interviewers must hear your contribution. Always quantify the Result ("cut inference cost 40%", "reduced p99 from 800 ms to 200 ms").

2. Live-coding strategy — the loop that prevents freezing

1. RESTATE the problem + ask clarifying Qs (inputs, constraints, edge cases)

4. CODE — narrate as you go; start with a brute force if needed
5. TEST — run through an example line by line; handle edge cases
6. OPTIMIZE — only after it works; state the new complexity

Q: you're stuck mid-problem — what do you do? → think out loud, state what you know, try a brute-force first ("let me get something working, then optimize"). Silence is the killer; interviewers can't help or score a blank screen. Q: brute force or optimal first? → a working brute force beats a broken optimal; verbalize the optimal and improve if time allows.

3. From-scratch ML implementations (the ML-coding round favorites)

Gradient descent for linear regression

import numpy as np
def fit_linreg(X, y, lr=0.01, epochs=1000):
    n, d = X.shape
    w, b = np.zeros(d), 0.0
    for _ in range(epochs):
        y_hat = X @ w + b                      # forward
        err = y_hat - y                        # residual
        w -= lr * (X.T @ err) / n              # dL/dw = Xᵀ(ŷ-y)/n
        b -= lr * err.mean()                   # dL/db
    return w, b

K-means (Lloyd's algorithm)

def kmeans(X, k, iters=100, seed=0):
    rng = np.random.default_rng(seed)
    centroids = X[rng.choice(len(X), k, replace=False)]
    for _ in range(iters):
        d = np.linalg.norm(X[:, None] - centroids[None], axis=2)  # (n,k) dists
        labels = d.argmin(axis=1)                                  # assign
        new = np.array([X[labels == j].mean(0) for j in range(k)]) # update
        if np.allclose(new, centroids): break                     # converged
        centroids = new
    return centroids, labels

Chapter 14 (Part II) — Tokenization & Context Windows

Part II — LLM & GenAI Deep Dive. Chapters 14–24 go deeper on the language- model stack, synthesized and de-duplicated from two reference interview handbooks. Where a topic was introduced in Part I (e.g. tokens in Ch 03), this part assumes that and pushes into the mechanics interviewers actually probe.

The token is the real unit of computation in an LLM — not the word, not the character. Cost, latency, context limits, and even subtle model behaviors all trace back to tokenization. It's a deceptively deep first question.

1. What a token is, and why not words

A token is a sub-word fragment from a fixed vocabulary (typically ~32k–128k entries). "gravity" might be one token; "antidisestablishmentarianism" splits into many; a rare emoji or a code identifier may fragment heavily.

Why sub-words instead of words?

  • Whole-word vocab would be enormous and still miss new/rare/misspelled words (out-of-vocabulary problem).
  • Characters make sequences far too long (compute is ~O(n²) in length).
  • Sub-words are the sweet spot: a bounded vocab that can still compose any string, common words stay one token, rare ones decompose gracefully.
import tiktoken
enc = tiktoken.get_encoding("cl100k_base")
enc.encode("tokenization")      # e.g. [3838, 2065]  -> 2 tokens, not 1 word
len(enc.encode("hello world"))  # 2
# rule of thumb (English): ~4 chars ≈ 1 token ≈ 0.75 words

Interview anchor: "Tokens are the unit the model counts, prices, and length-limits on — so I reason about tokens, not words, when estimating cost and context use."

2. BPE and SentencePiece — how the vocabulary is built

Byte-Pair Encoding (BPE): start from characters/bytes; repeatedly merge the most frequent adjacent pair into a new token; stop at the target vocab size. So frequent strings ("ing", "tion", "the") become single tokens, rare ones stay split. Byte-level BPE (GPT-2/3/4) operates on raw bytes, so it can encode any Unicode/binary input with zero true OOV.

SentencePiece: treats the input as a raw stream (spaces become a visible meta- symbol ▁), so it's language-agnostic and reversible — no reliance on whitespace tokenization. Preferred for multilingual models and languages without spaces (Japanese, Chinese). Supports BPE and unigram-LM algorithms.

Q: BPE vs WordPiece vs SentencePiece? → all are sub-word; BPE merges by frequency, WordPiece (BERT) merges by likelihood gain, SentencePiece is a framework that doesn't assume whitespace (good for multilingual). Q: how are OOV words handled? → there is no true OOV with byte-level schemes — unknown words just decompose into smaller known pieces (worst case, bytes).

3. Context window — the finite attention budget

The context window is the maximum number of tokens (prompt + generated output) the model can attend to at once — 8k, 128k, 1M depending on the model. It is a hard limit set by training (positional scheme + the O(n²) attention cost).

Key consequences:

  • Everything must fit: system prompt + history + retrieved context + the answer.
  • Bigger windows cost more (compute + KV-cache memory grows with length — Ch 22).
  • "Lost in the middle": models attend best to the start and end of a long context; mid-context facts are recalled worse — so put the most important material at the edges.

Q: context window vs training data? → the window is what the model sees now at inference; training data shaped the frozen weights. Q: does a 1M window mean I should stuff 1M tokens? → no — cost, latency, and "lost in the middle" degrade quality; retrieve the relevant slice (RAG, Ch 19) instead.

4. When input exceeds the window — truncation vs sliding vs summarization

Strategy What When
Truncation drop head or tail tokens cheap, lossy; logs, low-stakes
Sliding window process overlapping chunks, stitch long docs where local context suffices
Summarization / compression recursively summarize older context long chats; keep a running summary + recent turns
Retrieval (RAG) fetch only relevant chunks large corpora; the scalable answer

Q: a 200-page PDF won't fit — what do you do? → don't truncate blindly; chunk

  • embed + retrieve the relevant passages (RAG). For conversation length, keep a rolling summary plus the last N turns.

5. Special tokens — control signals in the stream

Models reserve tokens that aren't ordinary text: <bos>/<eos> (begin/end of sequence), padding, and chat-template markers that delimit roles (<|system|>, <|user|>, <|assistant|>, end-of-turn). The chat template is how a base model knows "this is the user speaking, now I respond."

Q: why do special tokens matter? → they're how the model is told structure (turn boundaries, stop conditions). Mismatching a model's chat template (wrong role markers) silently degrades quality — always format with the model's own template. Q: what is the <eos> token's job at inference? → emitting it is how generation stops (Ch 22); a missing/penalized <eos> causes run-on output.

6. Token economics — the production view

Tokenization is the cost/latency model:

  • Cost = input tokens × input price + output tokens × output price (output is usually 2–4× pricier). Fewer tokens = directly cheaper.
  • Latency ≈ prefill (process all input tokens, parallel) + decode (one token per step, serial) — so long outputs dominate latency (Ch 22).
  • Budgeting: reserve room for the answer (max_tokens), trim history, prefer retrieval over stuffing, and remember different models tokenize differently (the same text can be 20–30% more tokens on one tokenizer than another).
# Always budget: context_limit >= system + history + retrieved + max_output
def fits(enc, parts, max_output, limit):
    used = sum(len(enc.encode(p)) for p in parts)
    return used + max_output <= limit

Q: how do you cut LLM cost without changing the model? → shorter/templated prompts, retrieve instead of stuff, cap max_tokens, cache repeated prefixes, summarize history. Q: why does the same prompt cost differently across providers? → different tokenizers (token counts differ) × different per-token prices.

7. Interview Q&A (rapid fire)

  1. Q: why is the token the "unit of computation"? → the model embeds, attends over, and predicts tokens; length limits and billing are all per-token.
  2. Q: ~how many tokens is 1000 English words? → ~1300 (≈0.75 words/token).
  3. Q: why can code or non-English text blow up token counts? → they hit fewer high-frequency merges, so they fragment into more sub-word pieces.
  4. Q: what's the risk of a custom domain with lots of rare terms? → heavy fragmentation → more tokens → higher cost + less effective context; consider a domain tokenizer or fine-tuning.
  5. Q: truncation side effect to watch? → silently dropping the system prompt or the question itself; truncate the least important span deliberately.
  6. Q: why pad, and does padding cost compute? → batching needs equal lengths; padded positions are masked out but still occupy the batch (attention masks prevent them from affecting results).

  • ML Engineer: understand BPE/SentencePiece + OOV handling and how tokenizer choice affects training/eval.
  • MLOps Engineer: monitor tokens-per-request as a cost + latency signal (Ch 08), enforce budgets/limits.

9. Gotchas

  • Counting words instead of tokens when budgeting context/cost (§1, §6).
  • Ignoring the model's chat template / special tokens (§5).
  • Blind truncation that drops the question or system prompt (§4).
  • Assuming token counts are identical across models/providers (§6).
  • Stuffing a huge context and trusting mid-context recall ("lost in the middle", §3).

Chapter 15 (Part II) — Embeddings & Semantic Representations

Part II — LLM & GenAI Deep Dive. Ch 03 used embeddings as a black box for RAG. This chapter opens the box: what the vectors are, the geometry that makes similarity work, how to train your own, and the production decisions (dimension, quantization, thresholds, drift) that interviewers drill into.

An embedding is a learned map from text → a fixed-length vector where semantic similarity becomes geometric closeness. "doctor" and "physician" land near each other; "doctor" and "bicycle" land far apart. Everything in dense retrieval, clustering, classification, and recommendation rests on this one idea.

1. Concept map — three granularities

Embedding type Represents Made by
Token embedding one sub-word the model's embedding matrix (Ch 16)
Sentence/passage embedding a whole chunk's meaning pooling token states (mean/CLS) in a sentence-encoder
Document embedding a long doc pool/aggregate chunk embeddings, or store per-chunk

Token embeddings are internal to the transformer. Retrieval uses sentence/passage embeddings — a model fine-tuned so that semantically related texts get nearby vectors (Sentence-BERT, E5, GTE, OpenAI text-embedding-3, etc.).

Q: are an LLM's internal token embeddings the same as retrieval embeddings? → no. Internal token embeddings feed the transformer; retrieval embeddings are produced by a model trained with a similarity objective so distance is meaningful. Using raw mean-pooled LLM states for search usually underperforms a purpose-built embedding model.

2. The geometry: cosine vs dot product, and L2-normalization

Two vectors' similarity is measured by cosine (angle) or dot product (angle × magnitudes):

import numpy as np
def cosine(a, b):
    a, b = np.asarray(a), np.asarray(b)
    return a @ b / (np.linalg.norm(a) * np.linalg.norm(b))
  • Cosine ignores magnitude — it compares direction (meaning), which is what you usually want, since vector norm can vary with text length/frequency.
  • Dot product rewards larger norms too — fine only if norms are controlled.
  • L2-normalization (scale every vector to unit length) makes cosine and dot product identical, so you can use the faster dot product in your index while getting cosine semantics. Most vector DBs assume normalized vectors.
def l2norm(v): return v / np.linalg.norm(v)

Q: cosine or dot product? → cosine when norms vary or the model card says so; if you L2-normalize, they're equivalent — normalize and use dot for speed. Q: why normalize at all? → makes similarity reflect semantic direction, makes thresholds stable across items, and matches how most ANN indexes are tuned.

3. Hubness and anisotropy — why raw embeddings can retrieve badly

Two pathologies of high-dimensional embedding spaces:

  • Anisotropy: the vectors aren't spread evenly — they cluster into a narrow cone, so everything looks somewhat similar to everything, compressing the useful similarity range. Fixes: mean-centering, whitening, or using models trained with a contrastive objective (which spreads the space out).
  • Hubness: a few "hub" vectors show up as a nearest neighbor for far too many queries, crowding out the truly relevant results. Mitigations: normalization, hubness-aware scoring, better-trained embeddings.

Q: your dense retriever returns the same handful of docs for many different queries — what's happening? → likely hubness (and/or anisotropy). Check normalization, consider a better embedding model, or rerank (Ch 19) to correct first-stage bias.

4. Dense vs sparse, and bi-encoder vs cross-encoder

Dense (embeddings) capture meaning — they match paraphrases ("car" ≈ "automobile"). Sparse (BM25/TF-IDF, or learned sparse like SPLADE) capture exact terms — great for names, codes, rare keywords. They're complementary; hybrid retrieval fuses both (Ch 19).

The encoder distinction is the most-asked embedding question:

Bi-encoder Cross-encoder
Encodes query and doc separately query+doc together in one pass
Output a vector each → compare by cosine a single relevance score
Speed fast — docs pre-embedded & indexed slow — must run per (query, doc) pair
Use first-stage retrieval (recall over millions) reranking a shortlist (precision)

The mental model: the bi-encoder is the fast librarian who pulls 100 candidates; the cross-encoder is the careful reviewer who reads each one against the query and reorders the top 10.

Q: why not use a cross-encoder for everything? → it can't pre-compute — it'd have to score every (query, doc) pair at request time → O(corpus) per query, infeasible at scale. Bi-encoder for recall, cross-encoder for the final rerank.

5. Dimension, evaluation, and the cost/quality dial

  • Dimension (e.g. 384 / 768 / 1536 / 3072): higher dims usually capture more nuance but cost more memory and ANN time. Matryoshka embeddings let you truncate a vector to a shorter prefix and still retain most quality — a tunable cost/quality dial without re-embedding.
  • Evaluate before adopting: don't trust leaderboards blindly. Benchmark on your data with retrieval metrics — Recall@k, MRR, nDCG (Ch 19) — plus MTEB as a general prior. Measure latency and index size too.
# Quick offline check: does the right doc land in the top-k for known queries?
def recall_at_k(retrieved_ids, gold_id, k): return int(gold_id in retrieved_ids[:k])

Q: how pick an embedding model? → shortlist by MTEB + dimension/latency budget, then evaluate on your own labeled query→doc pairs; pick the best Recall@k/nDCG at acceptable cost. Q: bigger dimension always better? → no — diminishing returns, more memory/latency; Matryoshka truncation often gets ~95% of quality at half the dims.

6. Training custom / domain embeddings

General models miss jargon (legal, medical, internal product names). Options, cheapest first: (1) better prompts/instructions to the embedding model, (2) fine-tune on in-domain pairs, (3) train from a strong base.

The objective is contrastive — pull positives together, push negatives apart:

  • Contrastive / InfoNCE: for a query, its positive doc should score higher than a batch of negatives (softmax over similarities).
  • Triplet loss: (anchor, positive, negative) with a margin: max(0, d(a,p) − d(a,n) + margin).
  • Hard negatives are the key ingredient: negatives that are almost right (same topic, wrong answer) teach far more than random negatives. Mine them from top retrieval mistakes. Too-easy negatives → the model learns nothing useful; false negatives (actually relevant) → it learns the wrong thing.

Q: why do hard negatives matter so much? → random negatives are trivially separable, so the model plateaus; hard negatives sit near the decision boundary and force fine-grained distinctions — the single biggest lever in retrieval fine-tuning. Q: when fine-tune embeddings vs just rerank? → rerank first (cheap, no training); fine-tune embeddings when first-stage recall itself is the bottleneck on domain terms.

7. Long documents, multilinguality, and migration

  • Long docs: one vector can't represent a 50-page doc well → chunk and embed per chunk (Ch 19), or use late-interaction (ColBERT-style) / hierarchical pooling. Store chunk vectors with parent-doc metadata.
  • Multilingual: use a multilingual embedding model so "perro" and "dog" align; watch for one language dominating the space; evaluate per-language.
  • Index compression / quantization: Product Quantization (PQ) and scalar quantization shrink vectors (e.g. float32→int8) for big memory savings at a small recall cost — essential past tens of millions of vectors. Tune recall vs memory.
  • Migration is a re-index project: embeddings from model A are not comparable to model B's. Changing the embedding model means re-embedding the entire corpus and rebuilding the index — plan a dual-write/backfill + cutover, never mix vector spaces.

Q: you want to upgrade the embedding model in prod — what's the catch? → vectors aren't cross-compatible, so you must re-embed everything and rebuild the index; do it as a backfill to a new index, validate metrics, then cut over. Q: how set a similarity threshold? → empirically, on labeled pairs — pick the cutoff that hits your precision/recall target; thresholds don't transfer across models or even reindexes.

8. Monitoring & drift

Embedding systems rot like any model (Ch 08): incoming query/document distributions shift, new vocabulary appears, and retrieval quality silently decays. Track retrieval metrics over time, the distribution of top-1 similarity scores, and the fraction of "no good match" queries. Re-embed/refresh or re-fine-tune when drift crosses a threshold.

Chapter 16 (Part II) — Transformer Internals: Attention & Positional Encoding

Part II — LLM & GenAI Deep Dive. Ch 02b introduced the transformer and gave the one-line attention formula inside the training loop. This chapter is the deep version: the attention math step by step, why we scale by √d_k, what multiple heads buy you, the positional-encoding zoo (sinusoidal/RoPE/ALiBi), masking, the encoder/decoder variants, and the O(n²) problem + FlashAttention.

The transformer's one breakthrough: let every token look at every other token in parallel, weighting how much each one matters. No recurrence, no sequential bottleneck — which is exactly why it parallelizes on GPUs and replaced RNNs.

1. Self-attention — Query, Key, Value

Each token's embedding is projected into three vectors via learned matrices:

  • Query (Q): "what am I looking for?"
  • Key (K): "what do I offer / what am I about?"
  • Value (V): "the actual information I carry."

A token attends to others by comparing its Query to every Key (dot product → relevance score), softmax-normalizing those scores into weights, and taking the weighted sum of Values. The result is a new representation of the token that has mixed in context from the tokens it found relevant.

Attention(Q, K, V) = softmax( Q Kᵀ / √d_k ) V

import numpy as np
def attention(Q, K, V):
    d_k = Q.shape[-1]
    scores = Q @ K.T / np.sqrt(d_k)          # (seq, seq) relevance
    w = np.exp(scores - scores.max(-1, keepdims=True))
    w = w / w.sum(-1, keepdims=True)         # softmax → attention weights
    return w @ V                             # weighted sum of values

Q: what do Q/K/V do, in one breath? → Query asks, Key answers, Value carries the payload; attention weight = how well a token's Query matches another's Key, and the output is the Value-weighted mix. Q: where do Q/K/V come from? → three learned linear projections of the same input embeddings (in self-attention).

2. Why divide by √d_k — the scaling trick

As the head dimension d_k grows, raw dot products Q·K grow large in magnitude (variance scales with d_k). Large logits push softmax into saturated regions where one weight ≈ 1 and the rest ≈ 0 → vanishingly small gradients → training stalls. Dividing by √d_k keeps the logits at a stable scale so softmax stays in a well-conditioned range.

Q: why √d_k specifically? → dot-product variance grows ~linearly with d_k, so dividing by √d_k normalizes the standard deviation back to ~1, keeping softmax gradients healthy. Forgetting it is a classic "your attention won't train" bug.

3. Multi-head attention — many views at once

Instead of one attention with the full dimension, split into h heads, each with a smaller dimension, each with its own Q/K/V projections. Run attention in parallel per head, concatenate, and project back to d_model.

Why: different heads learn different relationships — one tracks syntax, another coreference ("it" → which noun), another local vs long-range patterns. One head can only express one weighting; multiple heads give the model several relational "lenses" simultaneously.

Q: why multiple heads instead of one big attention? → a single softmax produces one attention pattern; multiple heads capture different dependency types (syntactic, positional, coreference) in parallel, then combine them. Q: do more heads always help? → no — beyond a point heads become redundant (some can be pruned with little loss); it's a capacity/efficiency tradeoff.

4. Positional encoding — putting order back in

Attention is permutation-invariant: shuffle the tokens and the math is unchanged, because there's no notion of position in Q Kᵀ. So we inject position explicitly.

Scheme How Properties
Sinusoidal (orig. "Attention Is All You Need") fixed sin/cos patterns added to embeddings no parameters; limited length extrapolation
Learned absolute a trainable vector per position simple; cannot exceed trained length
RoPE (Rotary) rotate Q and K by an angle proportional to position encodes relative position; KV-cache friendly; extrapolates (esp. with scaling/YaRN) — the modern default (LLaMA, Mistral, Qwen)
ALiBi add a linear, distance-based bias to attention scores no parameters; strong length extrapolation, good for long context

Q: why do transformers need positional encodings? → without them, attention treats input as a set, not a sequence — "dog bites man" = "man bites dog". Q: why has RoPE largely won? → it expresses relative distances, plays well with the KV cache (you rotate keys as you go), and extends to longer contexts via RoPE-scaling/YaRN. Q: how extend a model's context window post-hoc? → RoPE scaling/YaRN, ALiBi-style biases, or sliding-window attention — re-fit or interpolate positions rather than retrain from scratch.

5. The rest of the block — FFN, residuals, LayerNorm

A transformer layer is attention + a feed-forward network (FFN), each wrapped with a residual connection and normalization:

x = x + Attention(LayerNorm(x))     # mix context across tokens
x = x + FFN(LayerNorm(x))           # transform each token independently
  • FFN (two linear layers + nonlinearity, often expands 4×): per-token processing — attention moves information between tokens, the FFN thinks about each token. Much of the model's parameters (and "knowledge") live here.
  • Residual connections let gradients flow through deep stacks (the x + shortcut) — without them, 96-layer models wouldn't train.
  • LayerNorm stabilizes activations; pre-norm (LN before the sublayer, shown above) is the modern, more stable choice vs the original post-norm.

Q: attention vs FFN — division of labor? → attention mixes information across positions; the FFN transforms each position independently and stores much of the learned knowledge. Q: why residuals + norm? → residuals enable gradient flow in deep nets (fight vanishing gradients, Ch 23); LayerNorm keeps activation scales stable so training converges.

6. Causal vs bidirectional masking, and the three architectures

  • Causal (masked) attention: each token may attend only to itself and earlier tokens (a triangular mask sets future scores to −∞). Required for autoregressive generation so the model can't "see the answer."
  • Bidirectional attention: every token sees the whole sequence — great for understanding (classification, retrieval embeddings), not for generation.
Architecture Attention Good for Example
Encoder-only bidirectional understanding/embeddings BERT
Decoder-only causal generation (LLMs) GPT, LLaMA
Encoder-decoder bi (enc) + causal (dec) + cross-attn translation, seq2seq T5

Q: why are modern LLMs decoder-only? → causal language modeling is a simple, scalable self-supervised objective over raw text, and decoder-only scales cleanly; it unifies "understand" and "generate" in one stack. Q: when encoder-only? → when you need a representation, not generation (embeddings, reranking, classification).

7. The O(n²) problem and FlashAttention

Attention compares every token to every token → compute and memory scale quadratically with sequence length n. Doubling context ~4×s the attention cost; the n×n score matrix is also a memory hog.

Mitigations:

  • FlashAttention: an exact-attention GPU kernel that processes the score matrix in tiles that fit in fast on-chip SRAM, never materializing the full n×n matrix in slow HBM → big speed + memory win, same result. FA-2/3 push parallelism further. This is why long context became practical.
  • Sparse / sliding-window / linear attention: approximate by limiting which tokens attend to which (e.g. local windows), trading some quality for sub- quadratic cost.

Q: why do transformers get expensive on long sequences? → attention is O(n²) in time and memory. Q: does FlashAttention change the result? → no — it's exact attention computed more efficiently (tiling + recomputation), reducing memory from O(n²) to O(n) and cutting HBM traffic.

8. Common failure modes (interviewers love these)

  • Attention sinks / collapse: heads dumping weight onto the first token or punctuation; some are benign, some hurt.
  • Position-related degradation past trained length (absolute schemes) → garbage beyond the window unless extended (RoPE scaling/ALiBi).
  • "Lost in the middle": weak recall of mid-context content (Ch 14).
  • Over-smoothing in very deep stacks: representations homogenize without good norm/residual design.

  1. Q: self-attention in one sentence? → every token builds a context-aware representation by taking a softmax-weighted sum of all tokens' Values, weighted by Query·Key similarity.
  2. Q: complexity of attention? → O(n²·d) time and O(n²) memory in sequence length (before FlashAttention's memory fix).
  3. Q: difference between self- and cross-attention? → self: Q,K,V from the same sequence; cross: Q from the decoder, K,V from the encoder (used in seq2seq).
  4. Q: what breaks if you drop the √d_k scaling? → softmax saturates, gradients vanish, training stalls.
  5. Q: RoPE vs ALiBi? → both encode relative position and extrapolate; RoPE rotates Q/K (KV-cache friendly, default), ALiBi adds a linear distance bias (very strong length extrapolation).
  6. Q: causal mask — what and why? → upper-triangular −∞ mask so a token can't attend to the future; required for autoregressive generation.
  7. Q: where does most of an LLM's "knowledge" sit? → largely in the FFN (MLP) weights, which dominate the parameter count.

Chapter 17 (Part II) — Pretraining Objectives & Model Families

Part II — LLM & GenAI Deep Dive. Ch 16 covered the architecture. This chapter covers what we train it to do (the objective) and the resulting taxonomy of models — the vocabulary interviewers use to place a model and pick the right tool.

A model architecture is inert until you give it a training objective — the self-supervised game it plays over raw text. The objective shapes what the model can do, which is why "autoregressive vs masked" is a foundational question.

1. What makes a language model "large"

A language model assigns probabilities to sequences of tokens — formally it models P(next token | previous tokens). "Large" refers to scale on three axes that move together: parameters (billions+), training data (trillions of tokens), and compute (GPU-years). Scale matters because of scaling laws — loss falls predictably as you grow data/params/compute — and emergence, where capabilities (in-context learning, reasoning, tool use) appear only past certain scales rather than being explicitly trained.

Q: what does "large" actually buy you? → not just lower loss but qualitatively new behaviors (few-shot in-context learning, instruction following) that small models don't show — emergence. Q: are bigger models always better? → no — for a fixed compute budget, the Chinchilla result says balance params and data (many early models were under-trained on data); and a small fine-tuned/distilled model often beats a huge one on a narrow task at far lower cost.

2. The two core objectives — autoregressive vs masked

Autoregressive (Causal LM) Masked LM (MLM)
Game predict the next token, left→right predict masked-out tokens from both sides
Attention causal (Ch 16) bidirectional
Strength generation understanding / representations
Example GPT, LLaMA, Mistral BERT, RoBERTa 
# Causal LM target = the input shifted by one (predict token t+1 from tokens ≤ t)
inputs  = tokens[:-1]
labels  = tokens[1:]        # cross-entropy over the vocab at each position (Ch 23)

# Masked LM: randomly mask ~15% of tokens; predict the originals
"the cat sat on the [MASK]"  ->  predict "mat"
  • Causal LM is the basis of all generative LLMs — a dead-simple objective that scales over any text and yields a model that can both understand and write.
  • MLM (BERT) sees the whole context, so it builds strong representations — great for classification/retrieval, but it can't generate left-to-right.
  • Next Sentence Prediction (NSP) was a secondary BERT objective (does sentence B follow A?). Historically important; later work (RoBERTa) showed it was largely unnecessary — a good "this looked reasonable but was dropped" interview note.

Q: why are generative LLMs autoregressive, not masked? → next-token prediction unifies understanding + generation in one scalable objective and lets the model produce arbitrarily long outputs; MLM is non-generative. Q: when is a masked/encoder model still the right pick? → when you need a representation (embeddings, reranking, classification), not generation (Ch 15).

3. Seq2seq and the encoder-decoder lineage

Sequence-to-sequence maps an input sequence to an output sequence (translation, summarization). The classic design is encoder-decoder (T5, BART): the encoder builds a bidirectional representation, the decoder generates while cross- attending to it (Ch 16 §6). RNN-based seq2seq came first but the transformer replaced it because attention removed the sequential bottleneck and the single fixed "context vector" that bottlenecked long inputs.

Q: why did transformers replace RNN seq2seq? → RNNs process tokens serially (slow, hard to parallelize) and compress the source into one vector (info bottleneck, poor on long inputs); attention sees the whole source in parallel and attends to any part. Q: decoder-only vs encoder-decoder for generation today? → decoder-only dominates general LLMs for its scaling simplicity; encoder-decoder still shines on fixed input→output transduction (translation, some summarization).

4. Foundation vs task-specific, generative vs discriminative

  • Foundation model: pretrained broadly on massive data, then adapted (prompt, RAG, fine-tune) to many tasks. Task-specific: trained/built for one job. The industry shifted to foundation models because one base amortizes across countless downstream uses.
  • Generative vs discriminative: generative models the data distribution and can produce samples (P(x) / P(x|y)); discriminative models the boundary P(y|x) for prediction only. An LLM is generative but can do discriminative tasks (classification) by generating the label (Ch 18).

Q: foundation vs task-specific tradeoff? → foundation = versatile, fast to adapt, but heavier and less specialized; task-specific = lean and optimal for the one task but brittle and costly to build per task. Q: is GPT generative or discriminative? → generative by training (next-token), but usable for discriminative tasks via prompting/constrained output.

5. LLMs vs classical statistical language models

Before neural LMs there were n-gram models: estimate P(word | previous n−1 words) from counts. They were simple and fast but: exploded combinatorially with context length, generalized poorly (unseen n-grams → zero probability, needing smoothing), and had no semantic understanding (purely surface counts). Neural LMs (and transformers) learn distributed representations that generalize across similar contexts and capture long-range, semantic dependencies.

Q: why did n-gram models hit a wall? → context is capped at n, the table grows combinatorially, and there's no generalization to unseen sequences or semantic similarity; neural LMs share statistical strength across related contexts via embeddings and attention.

Chapter 18 (Part II) — LLM Classification & Topic Discovery

Part II — LLM & GenAI Deep Dive. Two everyday production tasks: assigning known labels (classification) and discovering unknown structure (topic modeling). Ch 15 gave you embeddings; this chapter uses them and the generative LLM to do real text-analytics work, with the metrics and failure modes interviewers probe.

1. How a generative LLM does classification

A next-token model (Ch 17) can classify by generating the label — but the engineering is in making that reliable and machine-readable:

prompt = """Classify the ticket's intent as exactly one of:
[billing, bug, feature_request, other].
Return only the label.

Ticket: "I was charged twice this month."
Label:"""
# -> "billing"

Three ways to get a trustworthy class out:

  1. Constrained generation — restrict output to the allowed labels (JSON schema / logit bias / structured outputs) so it can't return free text.
  2. Log-probabilities — read the model's probability over each candidate label token → gives a usable confidence score, not just a hard label.
  3. Embedding + classifier head — embed the text (Ch 15) and train a cheap logistic-regression/SVM head (Ch 02) on top. Fast, calibratable, cheap at scale.

Q: how do you get a confidence score from an LLM classifier? → use the label tokens' log-probabilities (or predict_proba on an embedding+head). Free-text generation gives no calibrated confidence by itself. Q: how stop it inventing a label? → constrain output (enum/JSON schema/logit bias) and validate against the allowed set; reject/retry on violations.

2. Prompt vs fine-tune vs embedding-head — choosing the approach

Approach When Cost
Zero/few-shot prompt few labels, low volume, fuzzy/evolving taxonomy, fast to ship per-call $$
Embedding + small head high volume, stable labels, need calibrated probs + speed cheap inference, needs labeled data
Fine-tune the LLM nuanced labels, strict format/behavior, enough data (Ch 06) training + serving

Q: when prompt vs fine-tune for classification? → prompt for low volume / evolving labels / fast iteration; move to an embedding-head or fine-tune when volume makes per-call cost hurt or you need calibrated, low-latency, consistent output. Q: zero-shot vs few-shot? → zero = instruction + labels only; few-shot adds in-context examples (Ch 20) — helps most when labels are subtle or the desired format/edge-cases need demonstrating.

3. Designing the label taxonomy (the part candidates skip)

A classifier is only as good as its label set:

  • Mutually exclusive & exhaustive for single-label; include an other/ unknown escape hatch so the model isn't forced into a wrong bucket.
  • Clear, non-overlapping definitions — ambiguous labels create noisy labels and unstable predictions; write a one-line definition + example per label (doubles as the prompt and the annotation guide).
  • Keep the set as small as the problem allows; split only when a bucket is genuinely heterogeneous.

Q: model keeps confusing two labels — first thing to check? → the taxonomy: overlapping definitions. Tighten definitions, add disambiguating examples, or merge the labels. It's usually a label-design problem, not a model problem.

4. Class imbalance, multi-label, and metrics

  • Imbalance: rare-but-important classes (fraud, churn). Don't optimize raw accuracy (predicting "majority" scores high and is useless). Use class weighting, few-shot examples of rare classes, threshold tuning, and macro-averaged metrics that weight each class equally.
  • Multi-label (a ticket can be both "bug" and "billing"): predict an independent yes/no per label (sigmoid per label, not one softmax); evaluate with per-label precision/recall and micro/macro F1.
Precision = TP / (TP+FP)   "of predicted X, how many right?"
Recall    = TP / (TP+FN)   "of actual X, how many caught?"
F1        = harmonic mean(precision, recall)
Macro-F1  = unweighted mean of per-class F1   ← use under imbalance

Q: which metric for an imbalanced classifier? → precision/recall/F1 (and PR- AUC), macro-averaged, never plain accuracy. Pick the precision↔recall balance from the business cost of FP vs FN (Ch 09). Q: single- vs multi-label difference? → single-label = one softmax (pick one); multi-label = independent per-label decisions (sigmoid each) → different loss and metrics.

5. Confidence, human-in-the-loop, and failure modes

  • Confidence + routing: below a confidence threshold, escalate to a human or a stronger model. This is how you ship LLM classifiers safely on high-stakes decisions.
  • Production failure modes: label drift (new intents appear → need an other bucket + monitoring, Ch 08), prompt-format sensitivity, silent taxonomy mismatch, and over-trusting uncalibrated confidence.

Q: when add a human in the loop? → on low-confidence or high-stakes predictions (legal, medical, money), and to label the model's mistakes for the next training round. Q: classifier accuracy is fine offline but drops in prod — why? → distribution/intent drift or train/serve skew (Ch 08); new classes the taxonomy doesn't cover land in other or get misassigned.

6. Topic modeling vs classification — the unsupervised cousin

Classification assigns known labels. Topic modeling / clustering discovers structure when you don't know the labels — "what themes are in 1M support tickets?" No labels, no supervision.

The modern embedding-based pipeline (BERTopic-style) replaced classic LDA for most text:

1. EMBED every document (Ch 15)
2. REDUCE dimensions (UMAP) — high-dim embeddings cluster poorly (curse of
   dimensionality; distances concentrate)
3. CLUSTER (HDBSCAN: density-based, finds variable-shaped clusters AND an
   explicit "noise"/outlier bucket; or KMeans if you want a fixed k)
4. LABEL each cluster — pull top terms (c-TF-IDF) or ask an LLM to name it
5. EVALUATE + iterate

Q: why reduce dimensionality before clustering? → in high dimensions, distances concentrate (everything looks equidistant), so clustering degrades; UMAP projects to a low-dim space that preserves local structure, where HDBSCAN/KMeans work well. Q: why HDBSCAN over KMeans for topics? → it doesn't need a preset k, finds non-spherical clusters, and labels outliers as noise instead of forcing them into a topic — but KMeans is simpler when you truly want a fixed number.

7. Naming clusters, evolving topics, and evaluation

  • Naming for humans: raw clusters are useless to business teams — extract representative terms/documents and (often) have an LLM generate a short topic name + description. This is where LLMs supercharge classic topic modeling.
  • Evolving topics: topics drift over time (new products, events). Re-cluster on a schedule, or use online/streaming topic models; track topic prevalence over time.
  • Evaluation: topic quality has no single ground truth — use coherence (do a topic's top words co-occur sensibly?), separation between topics, human review, and downstream usefulness. Don't claim a clean accuracy number.

Q: how do you know your topics are "good"? → coherence scores + topic separation + human spot-checks + does it drive a decision; there's no labeled accuracy. Q: common topic-modeling mistakes at scale? → skipping dim reduction, forcing a fixed k, leaving clusters unnamed, ignoring outliers, and not handling topic drift over time.

Chapter 19 (Part II) — Retrieval & Production RAG (Deep)

Part II — LLM & GenAI Deep Dive. Ch 03 gave you the RAG pipeline at a high level. This chapter is the production version interviewers grade senior candidates on: hybrid retrieval + fusion, ANN index internals, reranking, query transformation, the offline/online metrics, multi-hop and agentic RAG, freshness, access control, caching, and when not to use RAG. Assumes Ch 03 and Ch 15.

1. Retrieval is two-stage and multi-signal

The naive RAG of Ch 03 (embed → top-k → stuff) breaks in production. Real systems combine signals and stages:

query → [rewrite/expand] → ┌ lexical (BM25)  ┐
                           ├ dense (vectors) ┤→ FUSE → rerank → top-N → prompt
                           └ metadata filter ┘   (cross-encoder)
  • Lexical (BM25) matches exact terms — unbeatable for names, IDs, codes, acronyms, rare jargon.
  • Dense (embeddings, Ch 15) matches meaning — paraphrases, synonyms.
  • Hybrid runs both and fuses the rankings; consistently beats either alone because they fail on different queries.

Q: lexical vs dense vs hybrid? → lexical = exact terms (precise on keywords/codes, blind to synonyms); dense = semantic (handles paraphrase, weak on exact rare tokens); hybrid fuses both. Use hybrid by default in production.

2. Score fusion — Reciprocal Rank Fusion (RRF)

You can't just add a BM25 score (unbounded) to a cosine score (0–1) — different scales. RRF fuses by rank, not raw score, so scales don't matter:

def rrf(rankings, k=60):                 # rankings: list of ranked doc-id lists
    scores = {}
    for ranked in rankings:
        for rank, doc in enumerate(ranked):
            scores[doc] = scores.get(doc, 0) + 1 / (k + rank + 1)
    return sorted(scores, key=scores.get, reverse=True)

Q: how do you combine BM25 and vector scores? → don't add raw scores (different scales); use RRF (rank-based) or learn weights / normalize first. RRF is the simple, robust default. Q: alternative? → weighted score normalization, or a learned fusion model, but RRF needs no tuning and is hard to beat.

3. Chunking — the highest-leverage RAG decision

How you split documents determines what can be retrieved:

  • Too large: chunks contain the answer plus noise → diluted embeddings, wasted context tokens, lower precision.
  • Too small: the answer gets split across boundaries → it's never fully retrieved.
  • Overlap (10–20%) prevents splitting a fact at a boundary.
  • Structure-aware / semantic chunking (split on headings, paragraphs, or semantic shifts) beats fixed-size character splits.
  • Small-to-big / parent-document: embed small precise chunks for matching but return the larger parent chunk for context.

Q: why is chunking so important? → it's what you actually retrieve; bad chunking caps the ceiling no reranker can fix. Match chunk size to your content and keep overlap. Q: a fact spans two chunks and never gets retrieved — fix? → overlap, semantic/structure-aware splitting, or small-to-big retrieval.

4. ANN index internals — HNSW vs IVF-PQ

At scale you can't compute exact nearest neighbors over millions of vectors per query → Approximate Nearest Neighbor (ANN):

Index Idea Trades
HNSW navigable small-world graph; greedy-descend layers great recall + speed; high memory
IVF cluster vectors; search only nearest cells (nprobe) tune speed↔recall via nprobe
PQ (Product Quantization) compress vectors into codes big memory savings, small recall loss
IVF-PQ cluster + compress scales to billions, memory-efficient

The universal knob is recall vs latency/memory: HNSW's efSearch, IVF's nprobe, PQ's code size. Tune to hit a recall target at acceptable cost.

Q: why approximate NN? → exact search is O(N·d) per query — infeasible past ~100k vectors; ANN gets ~99% recall at a fraction of the cost. Q: HNSW vs IVF-PQ? → HNSW for best recall/latency when memory allows; IVF-PQ when you must fit huge corpora in limited RAM. Metadata pre-filtering narrows the search first.

5. Reranking and query transformation

  • Reranking: first-stage retrieval optimizes recall (get the answer somewhere in top-50); a cross-encoder reranker (Ch 15) reads each candidate with the query and reorders to put the best in the top 3–5. Biggest single quality win after hybrid retrieval.
  • Query rewriting / expansion: the user's raw query is often a poor search key. Rewrite it (resolve pronouns from chat history, expand acronyms), generate multiple sub-queries, or use HyDE (have the LLM draft a hypothetical answer, embed that to retrieve) — drafts often match documents better than questions do.

Q: retrieval quality is mediocre — two highest-ROI fixes? → add a reranker and improve chunking/query rewriting, before touching the generator. Q: what is HyDE? → embed a hypothetical answer instead of the question; the answer's vocabulary aligns better with target passages.

6. Retrieval metrics — measure before you tune

Metric Measures
Recall@k is the gold doc in the top-k? (first-stage health)
MRR mean reciprocal rank of the first relevant hit
nDCG@k rank-weighted relevance (graded) — best overall ranking metric
Context precision / recall (RAG-specific) of retrieved context, how much is relevant / of relevant, how much retrieved

Q: which metric for first-stage vs final ranking? → Recall@k for the retriever (did we even fetch it?); nDCG/MRR for ranking quality after reranking. Build a labeled query→doc set and track these on every change.

7. Naive vs production RAG, multi-hop, agentic

  • Naive → production adds: hybrid + rerank, query rewriting, metadata filtering, citations, caching, access control, evals, and freshness handling.
  • Single-hop vs multi-hop: some questions need chained retrieval ("who is the CEO's spouse?" → find CEO → find spouse). Multi-hop retrieves, reasons, then retrieves again.
  • Agentic RAG: an agent (Ch 04) decides whether to retrieve, which tool/ index to hit, reformulates on weak results, and can loop — useful for complex, ambiguous, or multi-source questions; costs more, so gate it.

Q: when do you need multi-hop / agentic RAG? → when a single retrieval can't contain the full answer (compositional questions, cross-document reasoning) or the query needs interpretation/routing. Default to single-hop; escalate only when eval shows it's needed.

8. Grounding, citations, freshness, access control

  • Hallucination reduction: strict "answer only from context," citations to chunk IDs (validated post-hoc), retrieval-confidence thresholds, and an "I don't know / escalate" path (Ch 03 §5). Citations also make answers auditable.
  • Freshness: documents change → incremental indexing (upsert/delete on change), TTLs, and re-embedding on model upgrades (Ch 15 §7). Stale index = confidently outdated answers.
  • Permissions / access control: RAG can leak data the user shouldn't see. Enforce per-user access filters at retrieval time (filter by ACL metadata before the LLM sees a chunk) — never rely on the prompt to withhold retrieved content. This is a top production-security question.

Q: how do you stop RAG leaking restricted docs? → attach ACLs as metadata and filter by the requesting user's permissions during retrieval, so unauthorized chunks never enter the context. Prompt-level "don't reveal" is not a control. Q: how keep answers fresh? → incremental upsert/delete indexing + re-embed on model change; monitor for stale-answer complaints.

9. Caching, evaluation, and when NOT to use RAG

  • Caching: cache embeddings, frequent query→results, and even full answers for repeated questions — big cost/latency wins (Ch 22).
  • Eval (offline + online): offline = retrieval metrics + faithfulness/answer- relevance via RAGAS/LLM-judge on a golden set; online = resolution rate, thumbs-up, escalation rate (Ch 08 §6). "Eval-driven" RAG development.
  • When NOT to use RAG: the knowledge is small/stable enough to fit in the prompt; the task is reasoning/behavior not knowledge (fine-tune instead, Ch 06); or latency/cost budgets can't absorb retrieval. RAG adds moving parts — justify it.

Q: when would you not build RAG? → tiny/static knowledge (just prompt it), behavior/format problems (fine-tune), or strict latency budgets. RAG is for large, changing, citable knowledge. Q: how do you know your RAG is good? → offline retrieval + faithfulness metrics on a golden set, online resolution/CSAT — not vibes.

  1. Q: the production RAG stack in one breath? → rewrite query → hybrid (BM25+dense) retrieve with metadata filter → fuse (RRF) → cross-encoder rerank → grounded generation with citations → eval + cache.
  2. Q: biggest quality levers, in order? → chunking, hybrid+rerank, query rewriting, then the generator.
  3. Q: HNSW vs IVF-PQ tradeoff? → recall/latency vs memory footprint.

  5. Q: faithfulness vs answer-relevance? → grounded-in-context vs actually- answers-the-question; both needed (RAGAS).
  6. Q: RAG security #1 issue? → retrieval-time access control / data leakage.
  7. Q: multi-hop RAG? → iterative retrieve→reason→retrieve for compositional questions.

Chapter 20 (Part II) — Prompting, In-Context Learning & Orchestration

Part II — LLM & GenAI Deep Dive. Ch 03/04 used prompts and tool-calling inside RAG/agents. This chapter treats prompting as an engineering discipline: the message protocol, what makes a prompt reliable, in-context learning, structured outputs, prompt injection defense, versioning, and how to evaluate a prompt change like code.

Prompting is the cheapest, fastest way to steer an LLM — and the most under- engineered. Senior candidates treat prompts as versioned, tested artifacts, not strings someone tweaked once.

1. The message protocol — system / user / assistant / tool

Chat models take a list of role-tagged messages, not one blob:

messages = [
  {"role": "system",    "content": "You are a terse SQL assistant. Output only SQL."},
  {"role": "user",      "content": "Top 5 customers by revenue"},
  {"role": "assistant", "content": "SELECT ..."},          # prior turns = memory
  {"role": "tool",      "content": "<result of a tool call>"},
]
  • System: the durable contract — persona, rules, format, guardrails. Highest authority; set it deliberately.
  • User: the request.
  • Assistant: prior model turns (conversation memory).
  • Tool: results returned from function calls (Ch 04).

Q: what's the system message for, vs stuffing rules in the user turn? → the system message sets persistent, higher-priority behavior the model should honor across turns; user turns are per-request. Keep durable rules in system, the task in user. Q: how does a chat model "remember"? → you resend prior messages (history) each call; there's no server-side memory unless you build it (Ch 04).

2. What makes a prompt reliably good (not just verbose)

Reliability comes from specificity and structure, not length:

  • Role + explicit task + success criteria.
  • Output contract: exact format (JSON schema, allowed values), delimiters around inputs so the model can't confuse instructions with data.
  • Constraints and edge-case handling: "if X is missing, return null"; "if unsure, say you don't know."
  • Few-shot examples when format/behavior is subtle.
  • Decomposition: break a hard task into steps/sub-prompts rather than one mega- prompt.

Q: a prompt works 80% of the time — how do you harden it? → tighten the output contract (schema + validation), add few-shot examples of the failure cases, handle edge cases explicitly, delimit inputs, lower temperature, and add it to an eval set so the next change can't silently regress it. Q: longer prompt = better? → no — verbosity adds tokens/cost and can distract; precision and structure beat length.

3. In-context learning and few-shot

In-context learning (ICL): the model adapts to a task from examples in the prompt, with no weight updates — an emergent property of scale (Ch 17). This is why few-shot works at all.

  • Zero-shot: instruction only — fine for tasks the model already knows.
  • Few-shot: include 2–8 input→output examples — helps most when the format is unusual, the labels are subtle, or you need a consistent style. Diminishing returns (and token cost) past a handful; cover edge cases and class diversity in the examples.

Q: when does few-shot materially help? → when demonstrating an uncommon output format/style or disambiguating subtle labels; less so for tasks the model already does zero-shot. Q: why does in-context learning work without training? → large models learn to infer the task pattern from the prompt examples — an emergent capability; nothing is updated, the "learning" lives in the forward pass.

4. Chain-of-thought — and its product cost

Chain-of-thought (CoT): prompting the model to reason step by step before answering improves multi-step/reasoning tasks. But in a product:

  • It costs tokens and latency (more output) — don't enable it blindly on every call.
  • Exposed reasoning may be undesirable to show users; keep it hidden or use a reasoning model that does it internally.
  • For simple tasks it adds cost with no benefit.

Q: should you always use chain-of-thought? → no — it's a cost/latency tradeoff; use it for genuinely multi-step reasoning, not lookups or formatting. Modern "reasoning" models do it internally; you pay for reasoning tokens. Q: structured reasoning alternatives? → ask for a short rationale then the answer in a schema, or use self-consistency (sample several, majority vote) when accuracy justifies the cost.

5. Structured outputs & function calling

For anything a program consumes, free text is a liability. Force structure:

resp = client.chat.completions.create(
    model="gpt-4o", messages=msgs,
    response_format={"type": "json_schema", "json_schema": {
        "name": "ticket", "schema": {
            "type": "object",
            "properties": {"intent": {"enum": ["billing","bug","other"]},
                           "priority": {"type": "integer"}},
            "required": ["intent"]}}},
)
# then VALIDATE (pydantic) before trusting it
  • JSON mode / schema / grammar-constrained decoding guarantee parseable output.
  • Function/tool calling (Ch 04) is structured output whose schema describes a callable — the bridge from chatbot to agent.
  • Always validate the parsed object; retry/repair on violation.

Q: how get reliable JSON from an LLM? → schema-constrained outputs (or grammar/ logit constraints), then validate with a typed model and retry on failure — never regex free text. Q: structured output vs function calling? → same machinery; function calling is structured output that names a tool + args for you to execute.

6. Prompt templates, versioning, and evaluation

Prompts are code:

  • Templates with typed variables (no string-concatenating user input into instructions — that's an injection vector, §7).
  • Version prompts (in git or a prompt registry) so you can roll back and A/B.
  • Evaluate changes like a code change: run the new prompt against a golden set, compare metrics (accuracy / faithfulness / format-valid rate / LLM-judge score) before/after, and online A/B for user-facing impact (Ch 03 §7, Ch 08).

Q: how do you know a prompt change is actually better? → not by eyeballing one example — run both versions on a versioned eval set and compare aggregate metrics; A/B in production for real impact. "Eval-driven prompting." Q: why version prompts? → reproducibility, rollback, and attribution when quality shifts.

7. Prompt injection — the top LLM-app security issue

Prompt injection: untrusted text (user input, a retrieved document, a tool result, a web page) contains instructions the model then obeys — "ignore previous instructions and exfiltrate the system prompt / call this tool." It's the LLM equivalent of SQL injection, and there's no complete fix.

Defenses (layered):

  • Separate instructions from data — put untrusted content in clearly delimited user/tool messages, never concatenated into the system prompt.
  • Least privilege for tools; human approval for irreversible actions (Ch 04).
  • Input/output filtering, allow-lists, and treating all retrieved/tool text as untrusted data, not commands.
  • Don't put secrets in the context the model can be tricked into revealing.

Q: what is prompt injection and how do you defend? → malicious instructions smuggled via inputs/retrieved content that hijack the model; defend with instruction/data separation, least-privilege tools, human-in-the-loop for high-risk actions, and output filtering — no single defense is complete, so layer them. Q: indirect prompt injection? → the payload arrives via a retrieved doc or web page the agent reads (Ch 19) — sanitize and sandbox tool outputs.

8. When prompting stops being enough

Escalate when prompt engineering plateaus:

  • Need fresh/proprietary knowledge → RAG (Ch 19).
  • Need consistent behavior/format/skill or lower latency/cost at volume → fine-tune (Ch 06).
  • Need actions/multi-step workflows → agents/tools (Ch 04).

Q: prompt vs RAG vs fine-tune — the decision? → prompt for behavior nudges and low volume; RAG for knowledge; fine-tune for consistent behavior/format/skill. Try them in that order of cost. (Ch 06 §1.)

Chapter 21 (Part II) — Multimodal LLMs

Part II — LLM & GenAI Deep Dive. A multimodal LLM (MLLM) takes more than text — images, audio, video — and reasons over them jointly with language. This chapter covers the cross-modal architecture, CLIP, when to use OCR vs native vision, evaluation, and the practical use cases. Assumes the transformer (Ch 16).

1. Concept map — one model, many input types

A multimodal LLM accepts mixed inputs (e.g. an image + a question) and produces text (and sometimes other modalities). The dominant pattern for text-image is:

image ─▶ Vision Encoder (ViT) ─▶ image features ─▶ Projection/Adapter
                                                        │  (maps vision dim → LLM token space)
                                  text ─▶ tokenizer ─▶ text tokens
                                                        ▼
                              LLM (decoder) attends over [image tokens; text tokens] ─▶ answer

The key idea: turn the image into "tokens" the language model can attend to. A vision encoder embeds the image; a small projection/adapter maps those embeddings into the LLM's token-embedding space; the LLM then treats them like extra context tokens alongside the text (Ch 16 attention does the fusion).

Q: what's the common text-image architecture? → a pretrained vision encoder + a projection layer + a pretrained LLM; the projection aligns image features to the LLM's embedding space so attention can fuse the two. Often only the projection/adapter is trained while the encoder and LLM are largely frozen (parameter-efficient, Ch 06). Q: why not retrain everything? → reusing strong unimodal pretrained components + training only the bridge is far cheaper and works well.

2. CLIP and contrastive image-text alignment

CLIP was the pivotal idea: train an image encoder and a text encoder jointly so that a matching (image, caption) pair lands close in a shared embedding space and mismatched pairs land far — a contrastive objective (Ch 15 §6) over huge web image-text data.

Why it mattered:

  • It produced a shared image-text space enabling zero-shot classification ("is this a cat or a dog?" by comparing the image to text-label embeddings) and cross-modal retrieval (search images by text).
  • Many MLLMs use a CLIP-style vision encoder as their visual front-end.

Q: why is CLIP important? → it aligned vision and language in one embedding space via contrastive pretraining, enabling zero-shot image tasks and serving as the visual encoder behind many multimodal systems. Q: how does CLIP do zero-shot classification? → embed the image and each candidate label's text; pick the label whose text embedding is closest — no task-specific training.

3. Visual grounding

Visual grounding = the model's answer is actually tied to specific regions/ content of the image, not guessed from language priors. A grounded model can point to where in the image the evidence is (bounding boxes / region references), and its answers change correctly when the image changes. Ungrounded models "hallucinate from the caption distribution."

Q: what is visual grounding and why care? → it's whether the model's response is genuinely supported by image content (vs language priors). Strong grounding reduces visual hallucination and enables region-level answers (detection, "click the X"). Q: how test grounding? → perturb the image (remove/alter the object) and check the answer changes; or evaluate region/bbox accuracy.

4. OCR vs native vision-language understanding

Two ways to handle text inside images (documents, screenshots, receipts):

  • OCR pipeline: a dedicated OCR engine extracts text → feed the text to the LLM. Best for dense, structured text (contracts, tables) where you need exact character fidelity and cheap, deterministic extraction.
  • Native vision-language: the MLLM reads the image directly. Best when layout, visual context, diagrams, or sparse text + imagery matter together (a chart with a caption, a UI screenshot) and exact OCR isn't the bottleneck.

Often hybrid: OCR for the text layer + the MLLM for layout/semantics.

Q: OCR or native vision for document Q&A? → OCR when you need exact, dense text extraction (legal/financial docs); native VL when visual layout/diagrams carry meaning or text is sparse; hybrid for the best of both. Q: failure mode of native VL on dense documents? → it can misread small/dense text and numbers — a place where OCR is more reliable.

5. Multimodal prompting, evaluation, and failure modes

  • Prompting differs: you interleave images and text, can reference "the image above," and ordering/resolution matter; very high-resolution images get tiled into many tokens (cost ↑, Ch 14). Be explicit about what in the image to focus on.
  • Evaluation is harder than text: use task-specific sets (VQA accuracy, caption quality, document-QA exact match), grounding checks (§3), and human/LLM-judge for open-ended outputs (Ch 03 §7). There's no single metric.
  • Failure modes: visual hallucination (describing things not present), reading dense text/numbers wrong (OCR territory), spatial-reasoning errors (left/right, counting), bias from training data, and cost blow-ups from high-res tiling.

Q: how do you evaluate a multimodal system? → per-task metrics (VQA/doc-QA/ caption) + grounding/perturbation tests + human or LLM-judge for open-ended; never one number. Q: common MLLM failure modes? → visual hallucination, misreading dense text/numbers, weak spatial reasoning/counting, and high token cost on large images.

6. Audio and video — what changes

  • Audio: either transcribe (ASR/Whisper) → text → LLM, or a native audio encoder for tone/speaker/non-speech cues. Streaming + latency matter for voice.
  • Video: the big new axes are temporal (events across time) and scale — you can't feed every frame, so you sample frames (keyframes / fixed rate) and reason over them, trading coverage for cost. Long videos blow past context limits fast → sampling + retrieval over frames.

Q: how does video change the design vs static images? → temporal reasoning + massive token volume; you sample frames (you can't process them all) and may retrieve relevant segments, balancing coverage against context/cost. Q: audio approach? → transcribe-then-LLM for content; native audio encoder when prosody/ speaker/non-speech sound matters.

7. Which use cases pay off first

Highest business value tends to come from document intelligence (invoices, forms, contracts → structured data), visual support/QA (screenshots → triage), accessibility (alt-text,描述), and catalog/marketing (image tagging, description generation) — tasks with clear ROI and tolerance for human review. Open-ended "understand any image" demos look impressive but monetize slower.

Q: where would you deploy multimodal first? → document intelligence and visual support — concrete, measurable ROI with a human-in-the-loop safety net — before open-ended visual reasoning.

Chapter 22 (Part II) — Decoding & the LLM Inference Pipeline

Part II — LLM & GenAI Deep Dive. Ch 05 sized GPUs and serving at a high level. This chapter is the end-to-end forward pass an interviewer walks you through: prefill vs decode, the KV cache (with the memory math), sampling strategies, speculative decoding, the LM head, and streaming. The single most-asked systems-deep-dive for LLM roles.

1. The journey of one token (the 30-second story)

When you hit Enter:

text → TOKENIZE → EMBED (+ positions) → N transformer layers → LM HEAD
     → logits → SAMPLE one token → DETOKENIZE → stream to screen → repeat

Two phases that dominate every serving tradeoff:

  • Prefill: all input tokens processed in one parallel forward pass → builds the full KV cache. Lots of math per byte loaded → compute-bound.
  • Decode: generate one token per forward pass, attending to the cached keys/values. Tiny math, but it reloads the whole cache + weights from memory each step → memory-bandwidth-bound.

Q: prefill vs decode — why does the distinction matter? → prefill is compute-bound (parallel over the prompt), decode is memory-bound (one token at a time, dominated by moving the KV cache + weights). Almost every serving optimization targets the memory-bound decode phase. Q: what's TTFT vs TPOT? → time-to-first-token (prefill latency) vs time-per-output-token (decode speed) — UX cares about both; streaming hides TPOT.

2. The KV cache — and the memory math interviewers want

Each decode step's attention needs the Keys and Values of every prior token. Recomputing them every step is O(n²); instead they're cached after first computation. The catch: the cache grows with sequence length × batch and can exceed the model weights.

KV bytes ≈ 2 (K and V) × n_layers × seq_len × d_model × bytes_per_value

Worked example — a 70B model in FP16 (80 layers, d_model 8192, 4096 tokens):

2 × 80 × 4096 × 8192 × 2 bytes ≈ 10.7 GB     ← just the cache, per sequence,
                                               before the weights are even loaded

This is why long context is expensive and why concurrency is capped by memory.

Q: how big is the KV cache? → use 2·layers·seq·d_model·dtype; for a 70B/4k it's ~10 GB per sequence — it can dwarf activations and rival weights, and it's the real limit on how many concurrent requests fit. Q: how shrink it? → GQA/MQA (§3), KV-cache quantization (INT8/INT4), shorter context, and PagedAttention to eliminate fragmentation (Ch 05).

3. GQA and MQA — shrinking the cache by sharing K/V heads

Multi-head attention (Ch 16) has h query, key, and value heads — the cache scales with h. Two variants cut it:

Scheme KV heads Cache Quality
MHA (multi-head) h largest max expressiveness
GQA (grouped-query) h/g (shared across groups) ÷g ~MHA, tiny loss
MQA (multi-query) 1 (shared by all queries) minimum small quality hit

LLaMA-3, Mistral, Gemma use GQA — it lets big models serve more concurrent sequences (smaller cache) with negligible quality loss. The queries stay multi-head; only the K/V heads are shared.

Q: what problem do GQA/MQA solve? → the KV cache (a serving memory/bandwidth bottleneck), by sharing K/V across query heads so the cache shrinks ~g×. GQA is the sweet spot; MQA is most aggressive. Q: do they change the query heads? → no — queries remain multi-head; only K/V are grouped/shared.

4. From hidden state to logits — the LM head

After the final layer, the last token's hidden vector is projected by the LM head (a d_model × vocab matmul) into one logit per vocabulary entry, then softmax → a probability distribution over the next token.

The LM head is huge and hot: at d_model=4096, vocab=128k it's ~0.5B params and one of the most expensive ops in every step. Mitigations:

  • Weight tying — share the LM-head matrix with the input embedding matrix (saves memory).
  • Vocabulary pruning for narrow domains.
  • Speculative decoding (§6) amortizes the head cost across several tokens.

Q: why is the LM head expensive? → it's a (d_model × vocab) matmul over a 100k+ vocab every step — huge. Weight tying + speculative decoding reduce its cost. Q: logits vs probabilities? → logits are raw scores; softmax turns them into a probability distribution to sample from.

5. Sampling — choosing the next token

The distribution is set; how you pick controls quality:

  • Temperature (applied to logits before softmax, logits / T): T→0 = greedy (sharp, repetitive), T≈0.7 balanced, T>1 flatter/wilder. The global "creativity dial."
  • Greedy: always argmax — coherent but bland/repetitive; good for extraction/deterministic tasks.
  • Top-k: keep only the k highest-prob tokens, renormalize, sample — caps the long tail. k=1 = greedy; k≈50 typical. Fixed k doesn't adapt to confidence.
  • Top-p (nucleus): keep the smallest set of tokens whose cumulative probability ≥ p (≈0.9–0.95) — adapts: small nucleus when the model is confident, larger when uncertain. Has largely supplanted top-k for chat.
  • Repetition / frequency / presence penalties: down-weight already-used tokens to curb loops.
def sample(logits, T=0.8, top_p=0.9):
    logits = logits / T
    probs = softmax(logits)
    idx = probs.argsort()[::-1]                 # high→low
    cum = probs[idx].cumsum()
    keep = idx[:np.searchsorted(cum, top_p) + 1]  # nucleus
    p = probs[keep] / probs[keep].sum()
    return np.random.choice(keep, p=p)
  • Beam search: keep the top-b sequences (not tokens) and expand — maximizes total likelihood. Great for translation/closed-ended; bad for open-ended chat (bland, repetitive, expensive) — which is why sampling won there.

Q: temperature vs top-p? → temperature reshapes the whole distribution before sampling; top-p truncates it to a dynamic nucleus. They compose (set T, then nucleus-sample). Q: why isn't beam search used for chatbots? → it optimizes likelihood → safe, repetitive, low-diversity text, and costs b× compute; nucleus sampling gives more natural, varied output. Q: deterministic output? → T=0 (greedy) + fixed seed (Ch 03), still not 100% guaranteed across providers.

6. Speculative decoding — going faster without changing output

Decode is memory-bound (§1) — the GPU is starved one token at a time. Speculative decoding uses a small, fast draft model to propose several tokens ahead, then the big model verifies them all in one parallel forward pass, accepting the longest matching prefix and correcting at the first disagreement.

  • Net effect: multiple tokens generated per expensive big-model pass → 2–3× speedups with identical output distribution (the verification guarantees it).
  • Variants: a separate draft model, or self-speculation (Medusa/EAGLE-style extra heads).

Q: how does speculative decoding speed things up without hurting quality? → a cheap draft proposes k tokens; the target model verifies them in a single batched pass and accepts the agreeing prefix — so you amortize the costly forward pass / LM head across several tokens, and the accept/reject rule keeps the exact target distribution. Q: what limits the speedup? → the draft's acceptance rate; a poor draft gets few tokens accepted.

7. Detokenization, streaming, and serving throughput

  • Detokenize + stream: each sampled token id is converted back to text and sent to the client immediately — you see words appear before the model finishes, which is why TTFT/streaming dominate perceived speed.
  • Continuous batching (Ch 05): the engine swaps finished sequences out and new ones in every step, keeping the memory-bound GPU saturated — the biggest decode throughput lever.
  • Throughput vs latency: bigger batches raise tokens/sec across users but raise per-request latency; tune max_num_seqs against a p99 SLA.
  • Long-context serving is hard because the KV cache (§2) and prefill cost blow up with length → PagedAttention, GQA, cache quantization, and chunked prefill.

Q: why stream tokens? → it slashes perceived latency (first word fast) and is the natural fit for autoregressive generation. Q: single biggest decode throughput lever? → continuous (in-flight) batching, plus a smaller KV cache (GQA/quantization) so more sequences fit concurrently.

  5. Q: top-k vs top-p? → fixed cutoff vs adaptive nucleus; top-p preferred.

Chapter 23 (Part II) — Optimization & Math Foundations

Part II — LLM & GenAI Deep Dive. The math an interviewer reaches for to test whether you understand the model or just use it. Ch 02b covered the training loop and Ch 16 the attention formula; this is the focused reference for the recurring math questions — softmax, cross-entropy, gradients, KL, eigen/PCA, and why residuals/norm tame vanishing gradients.

You won't derive a transformer on a whiteboard, but you will be asked "why cross-entropy?", "what does the softmax do?", "what's KL divergence used for?" — crisp answers here separate levels.

1. Softmax — turning scores into a distribution

Softmax maps a vector of real logits to a probability distribution (positive, sums to 1):

softmax(z)_i = e^{z_i} / Σ_j e^{z_j}

It appears in two places: attention (normalizing Q·Kᵀ scores into weights, Ch 16) and the LM head (logits → next-token probabilities, Ch 22).

Numerical stability: e^{z} overflows for large z. Always subtract the max first — softmax(z) = softmax(z − max z) — which is mathematically identical but safe:

def softmax(z):
    z = z - z.max()          # stability, no change to the result
    e = np.exp(z)
    return e / e.sum()

Q: why subtract the max in softmax? → to prevent exp overflow; it's invariant to a constant shift, so the probabilities are unchanged. Q: what does temperature do to softmax? → dividing logits by T before softmax sharpens (T<1) or flattens (T>1) the distribution (Ch 22).

2. Why the dot product is in attention

Attention scores tokens by Q·K — the dot product measures alignment between two vectors (large when they point the same way). So "how relevant is token j to token i?" becomes "how aligned is i's Query with j's Key?" It's also a single, GPU-friendly matmul over the whole sequence. Then /√d_k rescales it (Ch 16 §2).

Q: why a dot product for attention scores? → it's a cheap similarity measure (alignment of Query and Key directions) that vectorizes into one matmul; high dot product = high relevance.

3. Cross-entropy — the language-modeling loss

Language modeling is classification over the vocabulary at each position, so the loss is cross-entropy between the model's predicted distribution and the true next token (a one-hot target):

L = − Σ_i y_i log(p_i)   =   − log(p_correct)   (since y is one-hot)

It directly maximizes the probability the model assigns to the actual next token. Perplexity = exp(cross-entropy) — the intuitive "how surprised is the model" metric (lower = better; a perplexity of 10 ≈ "as unsure as picking among 10 equally likely tokens").

import torch.nn.functional as F
loss = F.cross_entropy(logits, targets)   # fuses log-softmax + NLL, numerically stable

Q: why cross-entropy for LMs? → next-token prediction is multi-class classification; cross-entropy maximizes the log-likelihood of the true token and has clean gradients. Q: cross-entropy vs perplexity? → perplexity = exp(CE), a more interpretable scale; minimizing one minimizes the other. Q: why not call softmax then log then NLL separately? → use fused cross_entropy/log_softmax for numerical stability (avoids log(0)).

4. Gradients, the chain rule, and embedding updates

Backprop (Ch 02b) is the chain rule applied in reverse over the computational graph: the gradient of the loss w.r.t. any parameter is the product of local derivatives along the path from loss to that parameter. This is what makes training deep stacks tractable — gradients are computed once, reused, and composed.

Embedding gradients: an embedding row is just a lookup; in the backward pass only the rows for the tokens that appeared receive a gradient (the rest get zero that step) — so embeddings update sparsely, per observed token. That's why rare tokens train slowly (few updates).

Q: how do embeddings get trained? → the embedding table is a parameter; backprop sends a gradient only to the rows of tokens present in the batch (sparse update), nudging each token's vector to reduce the loss. Q: chain rule's role in backprop? → it lets you compute each parameter's gradient as a product of local derivatives along the graph, computed efficiently in one reverse pass.

5. Jacobian, eigenvalues, and PCA

  • Jacobian: the matrix of all first-order partial derivatives of a vector-valued function (each output w.r.t. each input). Backprop through a layer is effectively multiplying by (transposed) Jacobians; their scale governs whether gradients grow or shrink (§7).
  • Eigenvalues/eigenvectors: an eigenvector's direction is unchanged by a linear transform (only scaled, by the eigenvalue). PCA uses the top eigenvectors of the data covariance (largest eigenvalues = directions of most variance) for dimensionality reduction — keep the few directions that carry most of the information (relevant to embeddings, whitening, and UMAP's cousins, Ch 18).

Q: how do eigenvalues connect to dimensionality reduction? → PCA projects onto the eigenvectors of the covariance matrix with the largest eigenvalues — the axes of maximum variance — discarding low-variance directions. Q: what's the Jacobian intuition in deep learning? → it's the local linear map of a layer; products of Jacobians across layers determine gradient magnitude (vanish/explode).

6. KL divergence — where it shows up in LLMs

KL divergence D_KL(P‖Q) = Σ P log(P/Q) measures how one distribution differs from another (asymmetric, ≥0, 0 iff identical). In LLM training it's everywhere:

  • RLHF/PPO: a KL penalty keeps the fine-tuned policy close to the reference model so it doesn't reward-hack into degenerate text (Ch 06).
  • Knowledge distillation: minimize KL between the student's and teacher's output distributions (the student mimics the teacher's soft labels).
  • Label smoothing / variational methods also use it.

Q: where is KL divergence used in LLMs? → the RLHF KL penalty (stay near the reference policy), distillation (match the teacher distribution), and label smoothing. Q: is KL symmetric? → no — D_KL(P‖Q) ≠ D_KL(Q‖P); it's a divergence, not a distance.

7. ReLU, vanishing gradients, residuals & normalization

  • ReLU max(0, x): derivative is 1 for x>0, 0 for x≤0 — the constant-1 slope for positive inputs avoids the saturation that shrinks sigmoid/tanh gradients, which is why ReLU-family activations enabled deep nets. Dying ReLU: neurons stuck at ≤0 get zero gradient forever; fixes: Leaky/GELU, careful init, lower LR.
  • Vanishing/exploding gradients: in deep stacks, repeatedly multiplying small (or large) Jacobians (§5) makes gradients shrink toward 0 (or blow up).
  • Residual connections (x + f(x)) give gradients a shortcut path (the derivative of the identity is 1), so signal reaches early layers — the key enabler of very deep transformers.
  • Normalization (Layer/RMSNorm) keeps activation scales stable across layers, so gradients stay well-conditioned. Together, residuals + norm are why 96-layer models train at all (Ch 16 §5).

Q: why does ReLU's derivative matter? → it's 1 for positive inputs (no saturation), keeping gradients alive through depth — unlike sigmoid/tanh which saturate and vanish. Q: how do residuals fight vanishing gradients? → the identity shortcut passes gradient straight through (local derivative 1), so deep layers still receive a strong signal. Q: exploding gradients fix? → gradient clipping + normalization + careful init (Ch 02b).

  2. Q: why cross-entropy for LMs? → maximizes log-likelihood of the true next token; perplexity = exp(CE).
  3. Q: backprop in one line? → reverse-mode chain rule computing every parameter's gradient in one pass.
  4. Q: how do embeddings update? → sparsely — only rows of tokens seen that step.
  5. Q: PCA ↔ eigenvalues? → project onto top-eigenvalue eigenvectors of the covariance (max-variance directions).
  6. Q: KL divergence use in LLMs? → RLHF penalty, distillation, label smoothing; asymmetric.
  7. Q: residuals + norm — why? → preserve gradient flow + stabilize activations in deep nets.

Chapter 24 (Part II) — Advanced Architectures & Deployment Extensions

Part II — LLM & GenAI Deep Dive. The final chapter: the "senior signal" topics that come up in deeper rounds — Mixture-of-Experts, knowledge-graph augmentation, and the hard, unglamorous realities of shipping LLMs (bias, interpretability, privacy, the bottlenecks teams underestimate). Assumes Ch 05/ 16/19/22.

1. Mixture of Experts (MoE) — capacity without proportional compute

A dense model runs every parameter on every token. MoE replaces the FFN (Ch 16 §5) with many parallel "expert" FFNs plus a small router that sends each token to only the top-1 or top-2 experts:

token → router (softmax over experts) → pick top-k experts → weighted sum of their FFN outputs

So a model can have huge total parameters (capacity/knowledge) while only a small active subset runs per token (compute/cost). Mixtral, DeepSeek-MoE, and several frontier models use it.

Q: why is MoE attractive? → it decouples total capacity from per-token compute — you get the knowledge of a much bigger model at the inference cost of a small active fraction. Q: do you serve MoE cheaply because only k experts fire? → compute is small, but all experts' weights must be in GPU memory (you don't know which fire), so MoE is memory-heavy to serve even though it's compute-light. That memory tradeoff is the catch interviewers want.

2. MoE failure modes

The router is the fragile part:

  • Load imbalance: the router favors a few popular experts → others sit idle, wasting capacity. Fixed with an auxiliary load-balancing loss + capacity limits per expert.
  • Routing collapse: training degenerates so most tokens go to one expert (the model stops using the mixture).
  • Token dropping: when an expert hits its capacity, overflow tokens get dropped/ skipped → quality hit.
  • Training instability: the discrete routing decision is harder to train than a dense FFN; needs careful balancing-loss tuning.

Q: what new problems does MoE introduce? → load imbalance / routing collapse (a few experts dominate), token dropping at capacity, training instability from the routing, and high serving memory (all experts resident). Mitigate with load-balancing losses, capacity factors, and careful routing.

3. Knowledge graphs + LLMs (GraphRAG)

Vector RAG (Ch 19) retrieves similar text. A knowledge graph stores explicit entities + typed relationships ((Acme)-[:ACQUIRED]->(Beta)), which lets you answer multi-hop, relational, aggregate questions that flat retrieval struggles with: "which companies that our suppliers acquired are in the EU?" — a graph traversal, not a similarity search.

GraphRAG combines them: use the LLM to extract a graph from documents, then retrieve subgraphs (and/or do community summarization) as grounding context.

Q: when does a knowledge graph beat plain vector retrieval? → for relational/multi-hop queries, aggregations over connected entities, and when you need explicit, auditable structure (provenance, constraints). Vector RAG wins for fuzzy semantic lookup over unstructured text; many systems combine them. Q: downside of KGs? → costly to build and keep current; brittle to schema/entity- resolution errors.

4. A few more architecture notes

  • Adaptive softmax: speeds up the expensive LM-head softmax over huge vocabularies (Ch 22) by grouping tokens into frequency tiers — frequent tokens get full resolution, rare ones a cheaper path. A training/efficiency optimization for very large vocabularies.
  • GPT-style vs Claude-style ecosystems (developer view): both are decoder-only API LLMs; differences candidates cite are practical — context length, tool-use/ agent ergonomics, structured-output support, safety posture, pricing, and SDK/ prompt-format conventions. Choose per use case, not brand.
  • Hyperparameters beyond learning rate: batch size, warmup + LR schedule, weight decay, dropout, sequence length, gradient-clipping threshold, and (for serving) sampling params all matter; LR is the most sensitive but far from the only knob.

Q: GPT vs Claude for a developer — how decide? → on concrete needs (context window, tool/agent support, structured outputs, latency, cost, safety), validated on your eval set — not on brand loyalty. Q: which hyperparameters matter beyond LR? → batch size, LR schedule/warmup, weight decay, dropout, sequence length, grad-clip; interactions matter (e.g. LR × batch size).

5. Bias, interpretability, and privacy (the responsible-AI round)

  • Biased / systematically wrong outputs: sources are training data, prompt framing, and feedback loops. Mitigations: curated/debiased data, evaluation on slice/subgroup metrics (Ch 08), guardrails + filters, RLHF/constitutional methods, human review on sensitive paths, and red-teaming.
  • Interpretability is hard: LLMs are billions of entangled parameters; we have partial tools (attention maps, probing, sparse-autoencoder "features", influence functions) but no full explanation of why a token was produced — a real limitation to acknowledge in high-stakes domains.
  • Privacy: models can memorize and regurgitate training data; prompts/RAG context may contain PII that leaves your boundary via an API. Mitigations: PII redaction, data-residency/self-hosting, retrieval-time access control (Ch 19), differential-privacy training where feasible, and audit logging.

Q: how do you address biased model outputs? → measure it (subgroup/slice evals + red-teaming), then mitigate across the stack: data curation, guardrails, alignment, human-in-the-loop for sensitive decisions — and monitor in production. Q: why is interpretability hard for LLMs, and why care? → scale + entanglement mean no faithful full explanation; it matters for trust, debugging, compliance, and safety in regulated/high-stakes use.

6. Deployment bottlenecks teams underestimate

Beyond "serve the model" (Ch 05/22), the things that bite in production:

  • Evaluation & regression: without an eval harness, every prompt/model change is a gamble (Ch 03/20). The most underestimated investment.
  • Cost at scale: token costs compound; long contexts, retries, and agent loops explode budgets (Ch 04/22) — needs budgets, caching, model routing.
  • Latency tails (p99) under real traffic; rate limits / quotas on hosted APIs.
  • Data & prompt versioning, observability, and drift (Ch 08) — the MLOps tax.
  • Safety/abuse, prompt injection (Ch 20), and human-in-the-loop for irreversible actions.
  • Change management: models deprecate/upgrade; behavior shifts silently — pin versions and re-run evals.

Q: what deployment bottleneck do teams underestimate most? → evaluation + the full MLOps loop (versioning, monitoring, cost/latency control, drift, safety) — the model call is the easy 10%; making it reliable, cheap, and safe in production is the other 90%.
